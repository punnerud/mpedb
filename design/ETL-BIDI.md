# ETL-BIDI — reversibel ETL for PySpell (design-forarbeid)

**Status: prior-art-research (2026-07-16). Selve designet er ikke skrevet — dette dokumentet er
forarbeidet oppgave #52 krever før design + adversarial review, samme disiplin som DESIGN-DDL.**

Idéen (#52): mpedb lagrer allerede nok proveniens til at PG→mpedb→sqlite3→mpedb→PG er en rundtur
(mirror + type-proveniens). Generaliser det til brukerdefinert ETL via PySpell: registrer
funksjons-PAR som går begge veier, komponer dem til pipelines, og la databasen lagre det som
trengs for å reversere — slik at «tap» av data kan angres ved å kjøre pipelinen baklengs.

Formen er et *lens-par med residual*:

```
forward(x) → (y, residual)        inverse(y, residual) → x
```

Tapsfrie steg (roter bilde, legg til annotering) har residual = ∅. Tapsgivende steg
(farge→gråtone: residual = chroma; PNG→JPEG: residual ≈ original, så der vinner kilderetensjon)
deklarerer hva som lagres. Rundturen virker fordi det tapte LAGRES — samme mekanisme som
mirror-rundturen, aldri fordi mappingen er magisk inverterbar.

**Eksempel som binder #50 og #52 sammen (Morten, 2026-07-16) — versjonslagring som base + diff:**
to header-filer kan reflink-peke på samme blob (#50), men en redigering som skriver hele filen på
nytt koster 2× diskplass — og på ext4/macOS finnes reflink ikke i det hele tatt. Med toveis-ETL
lagres versjon 2 i stedet som base + diff: `forward(v1, edit) → v2` med *diffen som residual*,
`inverse(v2, diff) → v1`. DB-en verifiserer `apply(v1, diff) == v2` byte-identisk ved ingest
(forpliktelse 4), og lineage-tabellen vet hvilken base en diff hører til (forpliktelse 8).
Delta-motor-kandidater: `zstd --patch-from`, xdelta/VCDIFF (RFC 3284), bsdiff. Presedensen for
kjedene er git packfiles: delta-kjeder med *dybdegrense* og periodiske fulle snapshots — nøyaktig
Bennett-pebbling-knappen fra forpliktelse 6, i produksjon i tjue år. Logisk delta-komprimering er
FS-uavhengig — den virker der reflink ikke gjør, og de to komponerer: reflink for identiske
blokker, diff-residual for nær-identiske filer.

**Og containerformater som presentasjon (Morten, 2026-07-16):** en logisk fil kan *fremstå* som
.gz/.zip via mpedbfs (#54) mens mpedb lagrer innholdet i sitt eget format (delta-kjeder, extents,
dedup). To moduser med ulik pris: (a) **presentasjon** — syntetiser en fersk, deterministisk
gz/zip ved lesing (faste parametre, mtime=0); ingen residual, men bytene er ikke originalens.
(b) **rundtur** — en ingestet .gz/.zip pakkes ut til innhold + container-residual (gzip-header:
mtime/OS/FNAME; deflate-blokkstruktur; zip: central directory), og kan gjenskapes BYTE-IDENTISK.
Samme data har mange gyldige deflate-kodinger — det er *derfor* residualen trengs når originalens
hash/signatur betyr noe. Presedensen for (b) er skipped og gammel: Debians **pristine-tar/pristine-gz**
regenererer eksakte originale tarballer fra utpakket innhold + en liten delta — jbrd-mønsteret
anvendt på gzip, i produksjon i Debian-infrastrukturen i ~20 år. (`precomp`/`xtool` gjør det samme
for vilkårlige deflate-strømmer.) Zip-varianten gir en bonus: medlemmene ligger som logiske
rader/blober i mpedb (spørrbare, dedupede, delta-komprimerte), og arkivet er bare et view.

Research-spørsmålet var: **hva lærte alle som har bygget dette før oss?** Tre sveip:
akademisk BX/lens-teori, reversible programmeringsspråk, og industrielle systemer
(inkl. artefakt-korrelasjon i praksis). Kapitlene under er funnene; først destillatet.

## Designforpliktelser (destillert av all prior art)

Disse binder #52-designet. Hver av dem er betalt for av et konkret havari eller en konkret
suksess i kapitlene under.

1. **Formen er formelt fundert.** `forward(x) → (y, residual)` / `inverse(y, residual) → x` er
   nøyaktig symmetric-lens-komplementet (Hofmann/Pierce/Wagner), og komposittresidualen for en
   pipeline er *tuppelet av per-stegs-residualer, anvendt i revers* — komposisjon er mekanisk,
   ingen global analyse (Janus-egenskapen). Men residual-representasjoner er bare like opp til
   ekvivalens: sammenlign pipelines på observerbar rundtur-atferd, aldri på residual-bytes.

2. **Klassifiser hvert par eksplisitt**: bijektivt / inverterbart-med-residual / tapende-med-
   kilderetensjon. Cambria døde i den udeklarerte gråsonen; JPEG XL-encoderen NEKTER på input
   den ikke kan runde-trippe (`JXL_ENC_ERR_JBRD`) i stedet for å emittere et par som ikke
   inverterer. Fallback ved nekt = full kilderetensjon.

3. **Lovene som testes**: `inverse(forward(x)) == x` (GetPut-analogen) og residual-stabilitet
   `forward(inverse(y, r)) == (y, r)` (PutRes — fanger residual-drift). IKKE PutPut — samtlige
   praktiske systemer forkastet den. Rundtur testes *modulo deklarert canonizer* (quotient-
   lens-lærdommen: eksakt bytelikhet knekker på trivialiteter; byte-identisk er default-canonizer).
   `inverse(y, ∅)`-semantikken (kreasjonsstien) deklareres per steg eller nektes eksplisitt.

4. **Verifikasjon skalerer med hva du tør slette.** Beholdes kilden: samplet property-testing ved
   ingest. Slettes kilden: 100 % decode-og-bit-sammenlign FØR commit — Dropbox Lepton gjorde
   dette på 16 mrd. bilder og fanget en ikke-deterministisk buffer-overrun releasekvalifiseringen
   ellers hadde sluppet gjennom. Evidensen fra testing er statistisk, ikke universell — si det i docs.

5. **Residualen er eksplisitt og typet per steg.** Implisitt garbage akkumulerer gjennom
   komposisjon (RFun). Ancilla-skillet kopieres: read-only stegparametre (konfig, modeller)
   deklareres uendret og holdes utenfor residual-regnskapet (RFun v2/CoreFun, Sparcls ordinære
   argumenter). Branch-valg er residual-data: enten branch-tag i residualen, eller garantert
   disjunkte output-domener (first-match) — eksplisitt valg i steg-APIet.

6. **Lov aldri «minimal residual».** Minimalitet er uberegnbar (Glück/Yokoyama 2023); den harde
   nedre grensen er betinget entropi H(X|Y) — en stage som kaster k bits må lagre ≥ k bits.
   Tilby i stedet Bennett-pebbling som konfigurerbar knapp: checkpoints hver k-te stage mot
   rekomputeringstid ved revers (O(S·log T)-punktet er det teoretisk gunstige).

7. **Sparcls pin-regel som invariant:** alt en tapsgivende stage *leser* av pipeline-data skal
   gjenfinnes i (output, residual). Ingen tredje vei finnes.

8. **Artefakt-korrelasjon: content-hash + lineage-tabell, aldri filnavn.** Alle modne verktøy
   (DVC, Pachyderm, lakeFS, MLflow) landet der; posisjonell alignment var «a show-stopper»
   allerede i Boomerang. Lineage-rad: `(source_hash, output_hash, plan_hash, residual-ref,
   kjørings-ID)` — blake3 og content-hashede planer finnes allerede i mpedb. Én stabil
   kjørings-ID per pipeline-kjøring (Pachyderms global-ID-grep) gjør flertrinns-lineage til ett
   oppslag. Generert filnavnprefiks er kun UX-visning av lineage-oppslaget; embed en
   DerivedFrom-aktig ID (xmpMM-mønsteret) i output der formatet tillater. Hash-mismatch ved
   revers = eksplisitt feil («artefakt endret utenfor pipelinen»), aldri stille feil input.

9. **Residualformatet er et evighetsløfte.** Versjonér det fra dag én; decoder minimal,
   bounds-checked, `Corrupt`-aldri-panic (mpedb-regelen gjelder dobbelt). En ny versjon av et
   par som ikke kan konsumere gamle residualer er et NYTT par med ny hash (upcasting-regelen
   fra event sourcing). Lepton er arkivert, men 16 mrd. filer må dekodes for alltid.
   Pipelinens eget definisjonsformat versjoneres også — Cambrias prototype ble ubrukelig som
   system of record fordi *verktøyets* format driftet.

10. **Bygg ikke inverser der replay er gratis.** Mirror-lagets CDC er allerede kilderetensjon;
    reversible par forsvarer plassen sin når kilden skal slettes (Lepton-tilfellet) eller
    revers må være O(1) per artefakt i stedet for O(replay). Run-cache-idempotens følger
    gratis: `(input-hash, plan-hash) → output-hash` (DVC-grepet).

11. **Adopsjonsmønsteret:** eneste utplasserte lens-system (Augeas) vant ved smalt domene og
    dynamisk håndheving; alt med sterkere statiske ambisjoner ble værende forskningsspråk
    (Boomerang, Links, Sparcl). mpedbs valg — smal kontrakt håndhevet ved property-testing på
    ekte data — ligger på riktig side av det skillet, og selv Sparcl endte med runtime-assertions.

## Prior art: akademisk BX/lenses

### Asymmetriske lenses (Foster, Pierce et al., UPenn) og Boomerang

**Mekanisme.** En lens `l : C ⇌ A` er et par partielle funksjoner: `get : C → A` (projeksjon fra konkret kilde til abstrakt view) og `put : A × C → C` (fletter et oppdatert view tilbake — med den *opprinnelige kilden* som andre argument, siden `get` kaster informasjon som `put` må restaurere). Rammeverket ([Foster et al., TOPLAS 2007](https://www.cis.upenn.edu/~bcpierce/papers/lenses-toplas-final.pdf)) er en kombinator-DSL der hvert uttrykk denoterer begge retninger samtidig, og typede kombinatorer garanterer lovene komposisjonelt. Kreasjon (put uten original) håndteres med et eksplisitt "missing"-element Ω med defaults; Boomerang gjør dette til en tredje komponent `create : A → C` ([Bohannon et al., POPL 2008](https://www.cis.upenn.edu/~bcpierce/papers/boomerang.pdf)).

**Lover og hva som ble relaksert.**
- **GetPut**: `put (get c) c = c` (uendret view → uendret kilde). **PutGet**: `get (put a c) = a` (put må bevare *all* informasjon i viewet). Disse to = *well-behaved*, og forfatterne sier eksplisitt at å fjerne én av dem "significantly weakens the semantic foundation".
- **PutPut**: `put a' (put a c) = put a' c` (well-behaved + PutPut = *very well behaved*). **Denne ble droppet** fordi `map`, `flatten`, `merge` og conditionals bryter den "for reasons that seem pragmatically unavoidable". Kanonisk moteksempel: en versjonsteller-lens som inkrementerer ved endring oppfyller GetPut/PutGet men ikke PutPut. Boomerang-papiret er enda tydeligere: å kreve very-well-behavedness "would prevent writing many useful transformations" — sletter man en composer og legger ham inn igjen, kommer fødselsdatoene tilbake som defaults, "unfortunate, but the alternative is disallowing deletions!". Very-well-behaved ≙ Bancilhon–Spyratos' "constant complement" (TODS 1981) — det klassiske, for restriktive DB-regimet.
- **Quotient lenses** ([Foster/Pilkiewicz/Pierce, ICFP 2008](https://repository.upenn.edu/cis_papers/390/)): lovene er i praksis for sterke "on the nose" — de skal holde *modulo deklarerte ekvivalenser* (whitespace, feltrekkefølge, escaping). Mekanisme: *canonizers* og operatorer `lquot`/`rquot` som kvotienterer domene/kodomene; typen sporer hvilke ekvivalenser hver lens respekterer. Implementert som en utvidelse av Boomerang.
- **Totalitet er halve garantien**: uten totalitet er well-behavedness nesten triviell (enhver `get` kan pares med en `put` som bare er definert på `(get c, c)` — TOPLAS fotnote 3). Boomerangs typesjekker beviser totalitet via regulære uttrykk og entydighetssjekker (unambiguous concatenation/iteration) — mulig kun fordi domenene er regulære språk.

**Alignment-relaksering (dictionary/matching lenses).** Basic string-lenses aligner posisjonelt i `put`; ved reordering i viewet blandes data mellom rader (Copland/Britten-årstall byttes) — "a show-stopper for many of the applications we want to write". Løsning: *dictionary lenses* — brukeren markerer *chunks* (`match`) og en *key* per chunk; `put` parser kilden til skeleton + nøkkel-indeksert dictionary og henter chunks etter nøkkel, ikke posisjon. Nøkler trenger ikke være unike (kollisjoner matches posisjonelt). Den nye loven er **EquivPut**: `c ~ c' ⟹ put a c = put a c'` der `~` er nøkkel-respekterende reordering (*quasi-obliviousness*). [Matching lenses (ICFP 2010)](https://www.cis.upenn.edu/~bcpierce/papers/alignment.pdf) generaliserer: alignment-fasen skilles fra veve-fasen, og *vilkårlige heuristikker* kan plugges inn (posisjonell, minimal edit-distance, non-crossing/LCS, eller avledet fra selve operasjonen) — eneste krav for well-behavedness er at identiske argumenter gir identitets-alignment.

**Adopsjon.** Harmony-synkroniseringsverktøyet (bokmerker, adressebøker, kalendere) var drivkraften; Boomerang skippet ekte lenses for vCard, CSV, BibTeX/RIS, LaTeX, iTunes-bibliotek og SwissProt/UniProt. Den eneste brede utplasseringen av lens-ideene er **[Augeas](http://augeas.net/)** (Lutterkort, Red Hat 2007): asymmetriske get/put-lenses for Linux-konfigfiler, deployet via RHEL-verktøy, Puppets `augeas`-ressurs og libguestfs — symmetric lenses-papiret omtaler det som "commercial application". Augeas lyktes ved å snevre domenet (konfigfiler), beholde original-bevarende `put` (uendret tekst skrives tilbake ordrett) og *ikke* bevise lovene formelt per lens.

### Relational lenses (Bohannon/Pierce/Vaughan, PODS 2006; Links, Edinburgh)

**Mekanisme.** Et bidireksjonelt relasjonsalgebra-språk ([PODS 2006](https://www.cis.upenn.edu/~bcpierce/papers/dblenses-pods.pdf)): hvert uttrykk leses venstre-til-høyre som view-definisjon og høyre-til-venstre som update-policy. Primitiver: `select from R where P`, `join_dl R S` (suffikset er *policy-annotasjon*: slett fra venstre tabell ved view-sletting), `drop A determined by (X, default)`. Typesystemet bærer funksjonelle avhengigheter (FD-er, i "tree form") og record-predikater, og typereglene er teoremer som garanterer at lensen er well-behaved **og total** på det deklarerte skjemadomenet.

**Lover og relaksering.** Kun GetPut og PutGet kreves (deres Def. 3.2); PutPut er fraværende. Papiret viser konkret hvordan naive, symmetrisk-pene putback-definisjoner bryter lovene (deres `v⋈`-eksempel bryter både GetPut og PutGet), og at reparasjonene gir dokumentert *kontraintuitiv* atferd som er tvunget frem av PutGet: `select` sin putback **sletter** eksisterende kilderader som ville kollidert med FD-ene; `join_dl` krever at delte attributter er nøkkel i høyre tabell; `drop` krever at kolonnen er FD-bestemt og har deklarert default. De innrømmer at "totality imposes a stringent constraint on our lens design" — hovedbidraget var å finne brukbare totale domener via FD-er.

**Alignment.** Kjernen er *relational revision* `M ←_F L`: rader korreleres **nøkkelbasert via FD-er** (C-Match: matcher `m[X] = n[X]` → overskriv Y-feltene; ellers uendret) — aldri posisjonelt. Sletting vs. fravær skilles ved å simulere putback-get og fjerne rader som ville gjenoppstått (deres `L`-mengde) — "deleted vs. never-there"-problemet løses altså ved å regne ut hva viewet *skulle* inneholdt. Join-flertydighet løses ikke heuristisk, men ved at brukeren velger policy-variant (`join_dl`, `join_dr`, …) i syntaksen.

**Adopsjon.** PODS-papiret lovet en prototype; den kom aldri. [Horn/Perera/Cheney (ICFP 2018)](https://arxiv.org/abs/1807.01948) konstaterer at semantikken "has not been implemented or evaluated to date" og at relational lenses "have seldom actually been used" — den tilstandsbaserte `put` rekonstruerer hele kildetilstanden per oppdatering og er upraktisk. Første implementasjon kom 12 år senere i forskningsspråket **Links** (Edinburgh), og bare ved å gjøre semantikken *inkrementell* (delta-propagering, størrelsesordeners speedup over naiv put, uten å kreve updatable views i underliggende Postgres). Videreført i [Language-Integrated Updatable Views (2020)](https://arxiv.org/abs/2003.02191) og [Horns PhD (2022)](https://era.ed.ac.uk/handle/1842/39676). [Links selv](https://links-lang.org/) er og blir et forskningsspråk (akademiske case-studier som Covid-19-datakurering og database-wikier; ingen kjent industriell bruk).

### Symmetric lenses og edit lenses (Hofmann/Pierce/Wagner, POPL 2011/2012)

**Mekanisme.** En [symmetric lens](https://www.cis.upenn.edu/~bcpierce/papers/symmetric.pdf) `ℓ : X ↔ Y` består av en komplement-mengde `C`, et utpekt `missing ∈ C`, og `putr : X×C → Y×C`, `putl : Y×C → X×C`. Ingen side er primær: komplementet er konseptuelt `C_X × C_Y` — "privat informasjon" som kastes i hver retning. Lovene PutRL/PutLR sier at `putr(x,c) = (y,c')` medfører `putl(y,c') = (x,c')` (konsistente steady-state-tripler `(x,y,c)`). Mellomsteget i papiret er nøyaktig mpedb-signaturen: asymmetrisk lens med komplement har `get : X → Y×C`, `put : Y×C → X`, `create(y) = put(y, missing)` — dvs. `forward(x) → (y, residual)` og `inverse(y, residual) → x` er formelt identisk med komplement-formuleringen. Barbosa et al. sin reformulering gir også en tredje lov, **PutRes**: `res(put(v,c)) = c` (jf. [McKinna, Bx 2016](https://groups.inf.ed.ac.uk/bx/2016-Bx-CWC.pdf), som viser at komplement-elementer er *vitner* til konsistensrelasjonen — lovene er nettopp "hygienesjekken" på at operasjonene produserer gyldige vitner).

**Komposisjon og komplementer.** Ja — komposisjonen (Def. 4.2) er eksplisitt: `(k;ℓ).C = k.C × ℓ.C`, komposittkomplementet **er** tuppelet av per-stegs-komplementer, og put-ene tres gjennom stegvis. Men det avgjørende forbeholdet: algebraiske lover (assosiativitet, identitet) holder **kun opp til lens-ekvivalens** — `(j;k);ℓ` og `j;(k;ℓ)` er *ikke* samme lens "fordi komplementene deres er strukturert forskjellig"; ekvivalens defineres behavioralt (samme observerbare put-sekvenser, komplementer relatert via en relasjon R). Symmetriske PutPut-varianter vurderes og forkastes: "these laws appear too strong to be desirable in practice". Alignment forblir svakheten: deres liste-mapping er posisjonell, og innsetting først i listen gir "surprising (and probably distressing) results" — de peker på matching lenses eller deltaer som utvei.

**Edit lenses** ([POPL 2012](https://dl.acm.org/doi/10.1145/2103621.2103715)) er delta-utveien: edit-strukturer = monoide av edits `∂X` med partiell virkning på `X`; lensen er stateful monoide-homomorfismer `∂X × C → ∂Y × C` pluss en konsistensrelasjon `K ⊆ X × C × Y`; loven er at propagering av en definert edit bevarer `K`. Container-edits (insert/delete/modify/**rearrange**) bærer alignment-informasjonen *i selve editen* — retrospektiv diffing av tilstander bortfaller. Komposisjon kombinerer igjen komplementene som produkt.

**Adopsjon.** Ren teori: ingen implementasjon utover prototyper (Wagners [avhandling, 2014](https://repository.upenn.edu/edissertations/1488/)); innflytelsen gikk inn i delta-lenses/modelltransformasjonsmiljøet (Diskin, QVT-diskusjonene), ikke i produksjonssystemer.

### Lærdommer for mpedb

- **Verifiser lovene med property-testing, ikke statisk.** Alle systemene som beviste lovene statisk klarte det bare i trange domener (Boomerang: regulære språk med entydighetssjekk; relational lenses: FD-er i tree form — og forble uimplementert i 12 år). For vilkårlige brukerregistrerte funksjonspar er statiske garantier utilgjengelige; mpedbs runde-tur-test på ekte data ved ingest er dessuten nettopp *totalitets*-sjekken litteraturen viser at lovene er verdiløse uten (well-behavedness er nesten triviell for partielle lenses).
- **Test GetPut + PutGet; ikke krev PutPut.** PutPut ble forkastet av samtlige praktiske systemer ("pragmatically unavoidable" brudd i map/merge/conditionals; ≙ det for rigide constant-complement-regimet). Vurder derimot å teste **PutRes** som tredje lov: `forward(inverse(y, r)) == (y, r)` — at re-forward regenererer *samme residual* fanger residual-drift som ren `inverse(forward(x)) == x` ikke ser.
- **Rundeturen må testes modulo deklarert ekvivalens** (quotient lenses-lærdommen): eksakt bytelikhet knekker på whitespace/feltrekkefølge/encoding-trivialiteter. La hvert funksjonspar deklarere en valgfri canonizer, og test `canon(inverse(forward(x))) == canon(x)`; spor hvilken ekvivalens hvert steg respekterer i plan-metadataene.
- **Residual = komplement, formelt** — og komposittresidualen for en pipeline er tuppelet av per-stegs-residualer (symmetric lens-komposisjonsteoremet). Lagre residualer per steg og komponer mekanisk. Men: likhet mellom komposittkomplementer holder bare opp til ekvivalens — sammenlign pipelines på *observerbar atferd* (rundetur-resultat), aldri på residual-representasjonens bytes/hash.
- **Kreasjonsstien må være eksplisitt.** Hvert rammeverk trengte `Ω`/`missing`/`create` med defaults for "put uten original". mpedb må definere `inverse(y, ∅)`-semantikk per steg (deklarerte defaults), ikke la den være udefinert.
- **Artefakt-korrelasjon: nøkkelbasert, aldri posisjonell.** Posisjonell alignment var "a show-stopper" (Boomerang) og ga "distressing results" (symmetric lenses). Krev brukerdeklarerte nøkler per steg (dictionary lenses); relational lenses viser at FD-er/nøkler kan bæres i typene og at flertydige policyer (join-varianter) skal være eksplisitte annotasjoner, ikke heuristikk. Matching lenses viser at selve match-heuristikken trygt kan være pluggbar — eneste lovkrav er identitets-alignment på identisk input.
- **Propager edits, ikke tilstander.** Tilstandsbasert put skalerer ikke (relational lenses forble papir til delta-semantikken kom; edit lenses viser at edits bærer alignment gratis). mpedb har allerede CDC i mirror-laget — reversibel ETL bør konsumere endringsstrømmen, ikke diffe snapshots.
- **Adopsjonsmønsteret er en advarsel og en bekreftelse:** eneste utplasserte lens-system (Augeas) vant ved å snevre domenet, beholde original-bevarende put og bytte formelle bevis mot praktisk tekstbevaring; alt med sterkere statiske ambisjoner ble værende i forskningsspråk (Boomerang, Links). mpedbs valg — smal kontrakt (funksjonspar + residual) håndhevet dynamisk på ekte data — ligger på riktig side av det skillet.
## Prior art: reversible språk + artefakt-korrelasjon

### 1. Janus — reversibel imperativ (Lutz/Derby 1986; Yokoyama/Glück 2007)

**Mekanisme.** Hver statement er lokalt invertibel by construction: assignment finnes ikke, bare *reversible updates* `+=`, `-=`, `^=` der venstresidevariabelen ikke får forekomme i høyresiden (så oppdateringen er en bijeksjon på variabelen). Kontrollflyt bærer path-informasjon eksplisitt: `if` har både inngangstest og **exit assertion** (`fi`), løkker har **entry assertion** (`from`) + exit-test (`until`); baklengs kjøring bytter rollene. Informasjonsdestruksjon håndteres ved at den er forbudt uten bevis: globale variabler er null-initialisert, og `local x = e … delocal x = e'` krever at programmereren *oppgir verdien* variabelen har ved deallokering — en eksplisitt "jeg kan rekonstruere dette"-forpliktelse, sjekket som runtime assertion. Inversen er syntaktisk og lokal (statement-for-statement, blokker i revers), ingen history; språket er r-Turing-komplett (simulerer enhver reversibel TM). Kilder: [Janus (Wikipedia)](https://en.wikipedia.org/wiki/Janus_(time-reversible_computing_programming_language)), [PIRC/DIKU](https://topps.diku.dk/pirc/?id=janus), [Yokoyama & Glück 2007](https://dl.acm.org/doi/10.1145/1366230.1366239).

**Adopsjon.** Caltech-studentprosjekt 1982/86, gjenoppdaget og formalisert av Yokoyama/Glück 2007; playground-interpretere, undervisning ved DIKU, demoprogrammer (FFT, RTM-simulering). Ingen produksjonsbruk. Det ble værende akademisk fordi (i) hovedmotivasjonen (Landauer-grensen, reversibel/adiabatisk hardware) aldri materialiserte seg kommersielt, (ii) annoteringsbyrden er høy — hver sletting krever en delocal-assertion, hver branch en exit-assertion, og (iii) algoritmer må *redesignes* reversibelt, ikke bare oversettes ([oversikt, Wikipedia](https://en.wikipedia.org/wiki/Reversible_programming_language)).

**Lessons for mpedb.** (1) `delocal x = e'` er nøyaktig residual-kontrakten vår i imperativ form: destruksjon er lovlig bare når verdien kan angis/rekonstrueres — gjør destruksjon eksplisitt per stage, aldri implisitt. (2) Lokal invertibilitet gir gratis komposisjon: pipeline-inversen er stage-inversene i revers rekkefølge — ingen global analyse nødvendig; det er egenskapen mpedb bør bevare. (3) Janus' assertions er runtime-sjekker, ikke typer — vår property-testing av round-trips er samme designpunkt, og Janus viser at det holder for korrekthet men koster kjøretidssjekker på hver branch. (4) Full reversibilitet for *hele* språket var for restriktivt for reelle programmer — det er grunnen til at Sparcl og Eel finnes; ikke gjenta feilen ved å kreve at hele ETL-pipelinen er reversibel.

### 2. RFun / CoreFun — funksjonell reversibilitet, first-match, garbage

**Mekanisme.** RFun ([Yokoyama, Axelsen, Glück, RC 2011](https://link.springer.com/chapter/10.1007/978-3-642-29517-1_2)) er førsteordens og lineært: variabler brukes nøyaktig én gang (relevance — discard er destruksjon), konstruktorer er bijektive. Branching gjøres injektiv via **first-match policy**: forover vinner første matchende klausul; for at baklengs kjøring skal være deterministisk kreves at resultatet av klausul *i* ikke matcher resultat-mønstrene til tidligere klausuler — output-en selv koder hvilken branch som ble tatt, så ingen branch-tag må lagres. Duplikering er en eksplisitt operator hvis invers er equality check (dup ⇄ eq). Ikke-injektive funksjoner gjøres totale og injektive ved **ekstra output** ("garbage"): `plus :: Nat -> Nat <-> Nat` returnerer (bevart første argument, sum) — dette er nøyaktig vår residual. RFun v2/[rfun-interp](https://github.com/kirkedal/rfun-interp) la til **ancillae**: parametre som garantert er uendret over kallet og derfor ikke teller som residual. [CoreFun (Jacobsen, Kaarsgaard, Thomsen, RC 2018)](https://link.springer.com/chapter/10.1007/978-3-319-99498-7_21) ga et typesystem med unrestricted (ancilla) + relevant fragment som **statisk** sjekker first-match/injektivitet i mange tilfeller; resten forblir runtime-sjekker.

**Adopsjon.** Interpretere på GitHub, brukt i undervisning (20+ studenter, som tvang frem v2 med sukker og higher-order — [IFL'15-paperet](https://dl.acm.org/doi/10.1145/2897336.2897345) dokumenterer implementerings-lærdommene). Ingen bruk utenfor forskning.

**Lessons for mpedb.** (1) **Garbage vokser gjennom komposisjon**: naiv komposisjon g∘f akkumulerer både f-s og g-s garbage; RFun-litteraturens svar er Bennetts trick i funksjonell form — compute, kopier resultatet, *uncompute* (kjør f baklengs) for å slette mellomgarbage — som bytter residual-plass mot rekomputering. mpedb bør gjøre residualen **eksplisitt og typet per stage**, og la pipelinen velge: behold alle stage-residualer (rask invers) eller dropp intermediære og rekomputer (liten residual). (2) Ancilla-skillet er verdt å kopiere: konfig/parametre som passerer uendret gjennom en stage skal deklareres read-only og ikke havne i residualen. (3) Branch-diskriminering er et residual-spørsmål: hvis stage-output-domenene er disjunkte (first-match-egenskapen) trengs ingen branch-tag i residualen; ellers *må* tag-en inn. Gjør dette til en eksplisitt del av residual-designet per stage. (4) CoreFun viser at statisk verifikasjon av injektivitet bare dekker "mange tilfeller" — runtime-sjekk/property-testing forblir nødvendig selv med typesystem.

### 3. Sparcl — partiell invertibilitet som typenivå-egenskap (Matsuda/Wang, ICFP 2020 / JFP 2024)

**Mekanisme.** Sparcl ([ICFP'20-paper](https://mengwangoxf.github.io/Papers/ICFP20.pdf), [JFP 2024](https://www.cambridge.org/core/journals/journal-of-functional-programming/article/sparcl-a-language-for-partially-invertible-computation/809BDECF87B3748ED960FEFD42498BBE)) er lineært typet (λq-basert, som Linear Haskell) med typekonstruktoren **A• ("invertible")**: bare •-data er underlagt invertibel beregning; invertible funksjoner har type A• ⊸ B•. **Partiell invertibilitet uttrykkes ved vanlige argumenter**: `Int → Int• ⊸ Int•` er en familie av bijeksjoner indeksert av et ordinært (statisk) argument. Grensesnittet mellom verdenene er disiplinert: **pin : A• ⊸ (A → B•) ⊸ (A ⊗ B)•** lar en invertibel verdi brukes som statisk snapshot i irreversibel kode, men *bare* ved å beholde snapshotet i output-en (informasjonen kastes aldri); **lift : (A→B) → (B→A) → (A• ⊸ B•)** importerer et brukeroppgitt (f, f⁻¹)-par — *uverifisert*, Sparcl stoler på at paret er gjensidige inverser. Branching inverteres via **with-klausuler** (postconditions per branch) som sjekkes ved runtime som *eksklusive* assertions: forover feiler hvis branchens with ikke holder eller en annens også holder; baklengs velger with-ene branch. Lineariteten forbyr både discard og duplikering (dup eksplisitt; inversen er equality check). Garantien: `fwd e v = u` hvis og bare hvis `bwd e u = v` — total bijeksjon på faktisk domene/range.

**Adopsjon.** Prototype-interpreter ([github.com/kztk-m/sparcl](https://github.com/kztk-m/sparcl)), Agda-formalisering; motivert av serializers, Huffman/aritmetisk koding, LZ77, tre-rekonstruksjon. Forskningsspråk uten industriell bruk.

**Lessons for mpedb.** (1) Sparcls kjerneobservasjon er vår: **"partial-invertibility is the norm and bijectivity is a special case"** — realistiske pipelines har invertible deler parametrisert av irreversible deler (modellen i adaptiv kompresjon bygges likt begge veier). mpedb-design bør la stages ta statiske parametre som ikke inngår i residual-regnskapet. (2) pin-regelen er den viktige grensedisiplinen: en verdi som krysser fra reversibel til irreversibel kode må enten være statisk eller **bevares i output/residual** — formuler dette som en regel i ETL-BIDI: "alt en lossy stage leser av pipeline-data skal gjenfinnes i (output, residual)". (3) Sparcls lift er identisk med mpedbs brukerdeklarerte forward/inverse-par, og Sparcl kan heller ikke verifisere paret statisk — de *stoler*; mpedb property-tester i stedet, som operasjonelt er sterkere. Selv Sparcl valgte runtime-assertions (with-eksklusivitet) fremfor refinement types "for å holde typene enkle" — runtime-deklarerte residualer + property-testing er altså et legitimt designpunkt, ikke en fattigmannsløsning; prisen er at feil oppdages ved kjøring/test, ikke ved kompilering. (4) Typenivå-tilnærmingen kjøpte dem én ting testing ikke gir: *totalitet* av garantien (alle input, ikke bare testede) — vær ærlig i dokumentet på at property-testing gir statistisk, ikke universell, evidens.

### 4. Bennett — teoriens grenser for residualen

**Mekanisme/resultat.** [Bennett 1973](https://scispace.com/papers/logical-reversibility-of-computation-1kqufou0dk): enhver beregning kan gjøres reversibel via *compute–copy–uncompute* — kjør med history-tape, kopier resultatet, kjør baklengs så history-en forsvinner; garbage håndteres ved uncomputation, ikke sletting ([oversikt](https://en.wikipedia.org/wiki/Reversible_computing)). **Minimum residual**: for en total funksjon f over endelig domene er minimalt antall garbage-bits ⌈log₂ m⌉ der m = største preimage-størrelse max_y |f⁻¹(y)| ([Embedding of Large Boolean Functions, arXiv:1408.3586](https://arxiv.org/pdf/1408.3586)) — dvs. worst-case betinget informasjon; med variabel-lengde-koding er forventet residual-lengde nedad begrenset av H(X|Y). For programmer over uendelige domener er *minimal* garbage ikke beregnbart — det er ikke engang avgjørbart om en funksjon er injektiv (residual 0) ([Glück/Yokoyama, "Making Programs Reversible with Minimal Extra Data", New Generation Computing 2023](https://link.springer.com/article/10.1007/s00354-022-00169-z)). **Tid/plass-tradeoff**: full history er O(T) plass; Bennetts 1989-checkpointing (pebble game) gir O(S·log T) plass mot O(T^{1+ε}) tid; man kan gå helt til samme plass som irreversibelt, men da med eksponentiell tid ([Bennett 1989](https://mathweb.ucsd.edu/~sbuss/CourseWeb/Math268_2013W/Bennett_Tradeoffs.pdf), [Li/Vitányi pebble-analyse](https://arxiv.org/pdf/quant-ph/9703009)).

**Lessons for mpedb.** (1) **Ikke lov "minimal residual"** — det er uberegnbart i det generelle tilfellet; lov i stedet *deklarert, typet residual per stage* + testet round-trip. (2) Nedre grense er hard: en stage som destruerer k bits informasjon må lagre ≥ k bits residual — ingen smart koding omgår H(X|Y); dokumenter dette som forventningsstyring (en `DROP COLUMN`-stage har residual = kolonnen, punktum). (3) Bennett-pebbling er direkte anvendelig på pipelines: lagre checkpoints (fulle mellomresultater) hver k-te stage og rekomputer resten ved invers — residual-plass O(S log T) mot rekomputeringstid; gjør checkpoint-tetthet til en konfigurerbar knapp. (4) Compute–copy–uncompute er mønsteret for å hindre at residualer fra interne hjelpe-steg akkumulerer: interne mellomverdier bør uncomputes (droppes og re-deriveres), bare den *ytre* stage-residualen persisteres.

### 5. Korrelasjon av deriverte artefakter i praksis

**Mekanismer.** Fire konvensjoner i bruk: (1) **Deterministisk navning** — `foo.c → foo.o`, sidecar `foo.jpg`+`foo.xmp` (samme basename, annen extension), ffmpeg-batch med suffix/`%d`-mønstre. Enkelt, men OS/filbehandlere kjenner ikke koblingen, så rename/move av én av filene bryter den stille ([Sidecar file, Wikipedia](https://en.wikipedia.org/wiki/Sidecar_file)). (2) **Ekstern manifest/depfil** — gcc `-MD` .d-filer, provenance.json; path-nøklet og går stale når filer flyttes eller planen endres; ninja komprimerer dem inn i en intern database (`.ninja_deps`) nettopp fordi parsing av titusenvis av .d-filer dominerte (make bruker ~98 % av inkrementell tid på .d-parsing), men også deps-loggen får stale/feilaktige kanter når build-planen endres ([ninja manual](https://ninja-build.org/manual.html), [Fuchsia: How Ninja works](https://fuchsia.dev/fuchsia-src/development/build/ninja_how)). (3) **Embedded metadata** — XMP Media Management: `xmpMM:DocumentID` (stabil på tvers av versjoner), `InstanceID` (ny per lagring), `OriginalDocumentID` (roten), `xmpMM:DerivedFrom` (peker på forelderens ID-er) og `History` (event-array) ([Adobe xmpMM](https://developer.adobe.com/xmp/docs/xmp-namespaces/xmp-mm/), [exiv2 xmpMM](https://exiv2.org/tags-xmp-xmpMM.html)). Overlever rename/move fordi identiteten reiser *inne i* filen; feiler når formatet ikke kan embedde eller verktøy stripper metadata. (4) **Innholdsadressering** — identitet = content hash (Bazel/Nix-stil CAS, ninjas hash-baserte deps-DB): fullstendig robust mot rename, men krever ekstern database og re-hashing ved endring.

**Robust mønster** (konsensus på tvers av DAM og build-systemer): navn er *hint*, aldri identitet; identitet er embedded stabil ID der formatet tillater det **pluss** content hash i en ekstern database; provenance registreres i *begge retninger* (DerivedFrom-peker i den deriverte + manifest i DB), slik at én overlevende side kan rekonstruere koblingen.

**Lessons for mpedb.** (1) Nøkle residualer på **content hash av artefakten + stabil run/stage-ID**, aldri på filnavn/path — path-nøklede depfiler er den dokumentert skjøre varianten. (2) mpedb har en luksus DAM-verktøy mangler: databasen *er* den eksterne manifesten, transaksjonelt oppdatert sammen med residualen — utnytt det, men embed likevel en `DerivedFrom`-aktig ID i output-artefakter der formatet tillater (à la xmpMM), så koblingen overlever at filer forlater mpedbs kontroll. (3) Planlegg for stale-korrelasjon som normaltilfelle, ikke unntak (ninja-erfaringen): en invers som møter en artefakt hvis hash ikke matcher manifestet skal feile eksplisitt («artefakt endret utenfor pipelinen»), ikke produsere feil input.

### Lærdommer for mpedb (samlet)

- **Gjør residualen eksplisitt og typet per stage** — implisitt garbage akkumulerer gjennom komposisjon (RFun-erfaringen); komposisjonsregelen er at (residual₁, …, residualₙ) i revers rekkefølge inverterer pipelinen, som i Janus' lokale inversjon.
- **Destruksjon krever deklarasjon**: en stage som kaster informasjon må enten oppgi hvordan den rekonstrueres (Janus' `delocal e`) eller legge den i residualen (RFuns ekstra output, Sparcls pin) — ingen tredje vei finnes, per Bennett/H(X|Y).
- **Ikke lov minimale residualer** — minimalitet er uberegnbar (Glück/Yokoyama 2023); lov deklarert størrelse + testet round-trip. Nedre grense per stage er betinget entropi; si det rett ut i docs.
- **Skill ancilla fra residual**: read-only stage-parametre (konfig, modeller kjent i begge retninger) skal typemerkes som uendret og holdes utenfor residual-regnskapet (RFun v2/CoreFun, Sparcls ordinære argumenter).
- **Branch-valg er residual-data**: hvis en stage har flere kodeveier må residualen enten inneholde en branch-tag, eller stagen må garantere disjunkte output-domener (first-match/with-klausul-eksklusivitet) — gjør valget eksplisitt i stage-APIet, og la property-testeren sjekke disjunkthet.
- **Runtime-sjekk + testing er et legitimt designpunkt**: selv Sparcl håndhever branch-eksklusivitet ved runtime og stoler blindt på lift-par; mpedbs property-testing av forward/inverse-par er operasjonelt sterkere enn Sparcls trust, men svakere enn typegarantien — dokumenter at evidensen er statistisk.
- **Tilby checkpoint-knappen**: Bennett-pebbling → konfigurerbar avveining mellom residual-plass (lagre alle mellomresultater) og invers-tid (rekomputer fra checkpoints), O(S log T)-punktet er det teoretisk gunstige midtpunktet.
- **Artefakt-korrelasjon: hash + embedded ID, aldri filnavn** — path/navn-baserte koblinger (sidecars, depfiler) brytes ved rename/move; robust mønster er content hash i mpedb-manifestet pluss xmpMM-aktig `DerivedFrom`-ID embedded i artefakten der mulig, og eksplisitt feil ved hash-mismatch.
## Prior art: industrielle systemer

### Cambria (Ink & Switch) — composable lenses for skjemaevolusjon

**Mekanisme.** Cambria ([inkandswitch.com/cambria](https://www.inkandswitch.com/cambria/)) uttrykker relasjonen mellom to JSON-skjemaversjoner som en *edit lens*: én spesifikasjon oversetter data i begge retninger. Lenses er bygget av små komposable operatorer (`rename`, `convert`, `wrap`/`head`, `add`/`remove`, `hoist`, `in`) og komponeres til en graf der noder er skjemaversjoner; oversettelse mellom fjerne versjoner skjer via korteste vei gjennom grafen. Et sentralt valg: lenses transformerer *patches* (JSON Patch), ikke hele dokumenter — det passer systemer som utveksler inkrementelle endringer. Tidlige prototyper oversatte ved skrivetid; det viste seg skjørt (nye skjemaer lagt til etter at skrivinger var gjort, samtidige skrivinger under skjemaregistrering), så de landet på å lagre rå skrivinger i *writer schema* og oversette ved lesetid.

**Hva som skipet vs. ikke.** Ingenting nådde produksjon. Forfatterne er eksplisitte: «we do not pretend to have delivered a fully formed, production-quality solution», ytelse ble aldri målt, og prototype-issue-trackeren deres var «too unstable to be their only system of record» — primært fordi Cambrias eget lagringsformat endret seg underveis (lens-formatet trengte selv lensing). Dokumenterte åpne problemer: rekursive skjemaer, kryssdokument-migrering, og at `convert`-operatoren bryter lens-formalismen (utvikleren angir uavhengige frem/tilbake-avbildninger uten garantert konsistensrelasjon).

**Den viktigste innsikten** er trilemmaet deres: *consistency* (begge sider ser en meningsfullt ekvivalent verden), *conservation* (ingen side opererer på data den ikke kan se) og *predictability* (lokal intensjon bevares) kan ikke alle holdes når skjemaer divergerer — «there is no perfect option». Og det de manglet er nøyaktig residual-ideen: `{firstName, lastName}` → `{fullName}` kan ikke reverseres pålitelig, og «Cambria needs a way to express dependencies on other data sources» for å hente manglende data. Defaults ved lesetid (`add` fyller inn tomme arrays) fungerte derimot godt og flyttet defensive sjekker ut av applikasjonskoden.

**Lærdommer for mpedb:**
- Ikke lov total bidireksjonalitet. Klassifiser hvert funksjonspar eksplisitt: bijektivt / inverterbart-med-residual / tapende. Cambria døde i grenselandet der de lot som alt var inverterbart.
- Residual-mekanismen vår løser presist hullet Cambria dokumenterte — men bare hvis residualen fanges *ved forward-kjøring*, ikke rekonstrueres i etterkant.
- Lagre i writer-format og transformer ved lesing der det går; skrivetids-transformasjon binder deg til å re-migrere ved hver ny skjemaversjon.
- Pipelinens eget format (lens-/plan-definisjonene) trenger versjonering fra dag én — Cambrias prototype ble ubrukelig som system of record fordi *verktøyets* format driftet.

### JPEG XL jbrd + Dropbox Lepton — skipet residual i produksjon

**Mekanisme (JPEG XL).** JPEG XL rekomprimerer eksisterende JPEG-filer ~20 % mindre med byte-for-byte eksakt rekonstruksjon ([Wikipedia](https://en.wikipedia.org/wiki/JPEG_XL), [arXiv 2506.05987](https://arxiv.org/html/2506.05987)). Selve bildedataene re-kodes med bedre entropikoding; siden samme bildedata kan kodes på mange måter som JPEG-fil, lagres en separat **`jbrd`-boks** (JPEG bitstream reconstruction data) med alt som trengs for å skille kodingene: entropikoder, progressive scan-script, restart-markører, verdier av padding-biter og gjenværende app-markører ([libjxl format_overview](https://github.com/libjxl/libjxl/blob/main/doc/format_overview.md)). Rekonstruksjon = codestream + jbrd + metadatabokser (Exif/XMP/JUMBF). Residualen er «typically relatively small» — hundretalls byte (et dokumentert eksempel: 489 byte) mot ~20 % besparelse; og den er *valgfri*: bildet kan vises uten den, den trengs bare for eksakt fil-rekonstruksjon. Viktig: encoderen **nekter** på input den ikke kan runde-trippe (`JXL_ENC_ERR_JBRD`, [libjxl #2693](https://github.com/libjxl/libjxl/issues/2693)) i stedet for å produsere et par som ikke inverterer.

**Mekanisme (Lepton).** Dropbox' Lepton ([dropbox.tech](https://dropbox.tech/infrastructure/lepton-image-compression-saving-22-losslessly-from-images-at-15mbs), [NSDI-paper via acolyer](https://blog.acolyer.org/2017/05/01/the-design-implementation-and-deployment-of-a-system-to-transparently-compress-hundreds-of-petabytes-of-image-files-for-a-file-storage-service/)) gjorde det samme trikset (22 % via koeffisient-prediksjon + aritmetisk koder) på 16 milliarder bilder / flere petabyte, og *slettet originalene*. Verifikasjonsdisiplinen deres er gullstandarden: «All of our compression algorithms, including Lepton, decode every compressed file at least once and compare the result to the input, bit-for-bit, before persisting that file» — 100 % round-trip-sjekk, ikke sampling, med den komprimerte filen i «kernel-protected, read-only memory» under sammenligningen. I tillegg: hver release *kvalifiseres* ved round-trip over >1 milliard tilfeldige bilder før deploy — dette fanget en ikke-deterministisk buffer-overrun «after just a few million images». Determinisme-truslene var kompilator-UB og uinitialisert heap; forsvar var statisk linking og null-initialisering av all heap. Decoderen kjørte under seccomp (kun read/write på åpne fd-er). En produksjonshendelse med formatversjons-drift krevde skanning av milliarder av filer for å finne 18 som måtte re-enkodes.

**Hva som skipet.** Begge skipet reelt: Lepton i Dropbox-produksjon fra 2016 (senere [deprecated/arkivert](https://github.com/dropbox/lepton) — men filene må dekodes for alltid); JPEG XL er ISO/IEC 18181, i Safari 17 (2023), og tilbake i Chrome 145 (feb. 2026) via Rust-decoderen jxl-rs bak flagg ([The Register](https://www.theregister.com/2026/01/14/google_rekindles_relationship_with_jilted/)).

**Lærdommer for mpedb:**
- Byte-identitet er en *verifikasjonsdisiplin*, ikke en egenskap ved funksjonsparet. Skal kilden slettes etter konvertering, må verifikasjonen være 100 % ved ingest (billig: både x og y er i minnet). Sampled property-testing er bare tilstrekkelig når kilden beholdes.
- Determinisme på tvers av versjoner er den skjulte kostnaden: `inverse` må gi samme x om tre år, på annen CPU, etter re-kompilering. Lås transformasjons-identiteten til residualen — mpedb content-hasher allerede planer; plan-hashen hører hjemme i lineage-raden.
- Residualen blir liten når forward bevarer mesteparten av informasjonen (hundretalls byte mot 20 % gevinst), men encoderen må *detektere* ikke-inverterbar input og falle tilbake til full kilderetensjon — aldri emittere et par som ikke runde-tripper.
- Et residualformat er et evighetsløfte: Lepton er arkivert, men 16 mrd. filer må fortsatt dekodes. Versjonér residualformatet og hold decoderen minimal og bounds-checked (mpedbs «Corrupt, never panic»-regel gjelder dobbelt her).

### DVC / Pachyderm / lakeFS / MLflow — kilderetensjon + lineage i stedet for inverser

**Mekanisme.** Alle fire valgte å *beholde kilden* og korrelere via identiteter, ikke å invertere transformasjoner. **DVC** ([internal files-doc](https://doc.dvc.org/user-guide/project-structure/internal-files)) bruker content-addressable cache: MD5-hash av innholdet gir stien (`.dvc/cache/files/md5/ec/1d29…`), som deduper identiske filer uansett navn; små `.dvc`-pekefiler og `dvc.lock` i git binder outputs til dependencies via *hashene deres*, og run-cachen nøkler kjøringer på (eksakt dependency-innhold + kommando). **Pachyderm** ([Global ID-doc](https://docs.pachyderm.com/latest/concepts/advanced-concepts/globalid/)) gir hvert nytt commit en ID som *alle* nedstrøms commits og jobber i DAG-en deler — output-commit ↔ input-commit-korrelasjonen er dermed sann ved konstruksjon, eksponert i pipelinen som `PACH_OUTPUT_COMMIT_ID` / `<input>_COMMIT`. **lakeFS** ([versioning internals](https://docs.lakefs.io/v1.66/understand/how/versioning-internals/)) bygger et to-lags Merkle-tre av content-adresserte SSTable-«ranges» (Graveler); diff mellom commits koster proporsjonalt med endringen, og lineage på tvers av repos gjøres via commit-metadata som lenker kodeversjon til dataversjon. **MLflow** ([tracking-doc](https://mlflow.org/docs/latest/ml/tracking/)) skiller backend store (metadata-DB) fra artifact store (blobs); artefakter nøkles `run_id` + artifact-sti, kildekode via taggen `mlflow.source.git.commit`, og input-datasett via `mlflow.log_input` med Dataset-digest/hash.

**Hva som skipet.** Alle fire er i bred produksjonsbruk (Pachyderm kjøpt av HPE; DVC de facto-standard i ML-repoer; lakeFS på petabyte-lakes). Det som *ikke* finnes hos noen av dem: inverse transformasjoner. Reproduksjon gjøres alltid ved å kjøre forward på nytt fra den retenerte kilden — inversjon ble aldri engang forsøkt som produktfunksjon.

**Konvensjonen som vant korrelasjonsspørsmålet:** content-hash + lineage-tabell, overalt. Filnavnkonvensjoner er *ingen steders* primærnøkkel — de ryker ved rename, kopi og dedup. Sidecar-filer finnes (DVCs `.dvc`), men de er bare *bæreren* av content-hashen, versjonert i git; nøkkelen er hashen. Pachyderm og MLflow viser tillegget: en delt, stabil kjørings-ID (global commit ID / run_id) som binder *hele* settet av inputs, transformasjonsversjon og outputs i én lineage-rad.

**Lærdommer for mpedb:**
- Korrelér output↔kilde via en lineage-tabell nøklet på content-hash (blake3 finnes allerede i mpedb), med rad `(source_hash, output_hash, plan_hash, residual-ref, kjørings-ID)`. Generert filnavnprefiks er ute; sidecar bare som transportform for hashen.
- Én stabil ID per pipeline-kjøring (à la Pachyderms global ID) gjør flertrinns-lineage triviell å spørre — ikke rekonstruer kjeden fra parvise rader.
- DVCs run-cache-idé overføres direkte: (input-hash + plan-hash) → output-hash gir gratis idempotens ved re-ingest av samme fil.
- Skill metadata-lager fra blob-lager (MLflow-splitten): lineage-rader og residualer i mpedb-tabeller; store kildeblobs kan bo eksternt, referert ved hash.

### Event sourcing / CDC — reversibilitet ved replay

**Mekanisme.** I event sourcing ([Fowler](https://martinfowler.com/eaaDev/EventSourcing.html)) er loggen selve kilderetensjonen: «We can discard the application state completely and rebuild it by re-running the events from the event log on an empty application.» Reversering gjøres enten ved snapshot + replay forover, eller ved at hver event bærer nok til å reverseres — «storing the previous values on any value that is changed, or by calculating and storing differences on the event». Det siste *er* residual-ideen, uttalt av Fowler: eksplisitt reversering lønner seg først «when reversing a few events is much more efficient than using forward play on a lot of events». Dokumentert smerte: eksterne sideeffekter under replay, og event-versjonering — Greg Youngs regler ([Versioning in an Event Sourced System](https://github.com/luque/Notes--Versioning-Event-Sourced-System)) er at events er uforanderlige fakta, en ny versjon må være konverterbar fra den gamle (ellers er det en *ny* event), oppgradering skjer ved lesetids-*upcasting*, og omskrivninger gjøres ved å transformere hele strømmer til nye strømmer — aldri redigere på stedet.

**Lærdommer for mpedb:**
- Replay-fra-kilde slår inverser+residualer når kilden uansett beholdes — og mpedbs mirror-lag *er* allerede en retenert CDC-logg. Inverser+residualer forsvarer plassen sin i nøyaktig to situasjoner: kilden skal slettes (Lepton-tilfellet), eller reversering må være billig og lokal (én rad, ikke O(logg-lengde) replay).
- Fowlers «store previous values on the event» viser at residualen bør fanges *i commit-øyeblikket* av forward-transformasjonen, i samme transaksjon — ikke beregnes ved behov.
- Upcasting-regelen overført: en ny versjon av et funksjonspar må kunne konsumere gamle residualer, ellers er det et *nytt* par med ny plan-hash — og gamle residualer må beholde sin gamle decoder (jf. Lepton-evighetsløftet).

### Lærdommer for mpedb (samlet)

- **Verifikasjon skalerer med hva du tør slette.** Beholdes kilden: sampled round-trip-testing ved ingest er nok (à la Leptons release-kvalifisering over 1 mrd. bilder). Slettes kilden: 100 % decode-og-bit-sammenlign før commit, som Lepton — sampling er ikke forsvarlig.
- **Content-hash + lineage-tabell vant artefakt-korrelasjonen i samtlige modne verktøy** (DVC MD5-CAS, Pachyderm global commit-ID, lakeFS Merkle-ranges, MLflow run_id + dataset-digest); filnavnkonvensjoner er ingen steders primærmekanisme fordi de ryker ved rename/kopi/dedup. Sidecars er kun transport for hashen.
- **Klassifiser funksjonsparene eksplisitt** (bijektiv / inverterbar-med-residual / tapende) og la encoderen nekte på ikke-inverterbar input med fallback til full kilderetensjon (jbrd-modellen) — Cambrias udeklarerte gråsone er der lens-tilnærmingen brøt sammen.
- **Bind residualen til transformasjons-identiteten.** Lagre plan-hash (mpedb har content-hashede planer) i lineage-raden; determinisme på tvers av kompilering/arkitektur er den skjulte kostnaden ved byte-identitet, og versjonsdrift skjedde selv hos Dropbox.
- **Residualformatet er et evighetsløfte:** versjonér det, hold decoderen minimal, bounds-checked og `Corrupt`-aldri-panic, og planlegg for at gamle residualer må dekodes lenge etter at paret er avviklet.
- **Ikke bygg inverser der replay er gratis:** mirror-lagets CDC er allerede kilderetensjon; reversible par hører hjemme i konverteringsflyten der originalen slettes eller reversering må være O(1) per rad.
## Prior art: inverterbare kombinator-DSL-er og statisk verifisering

### 1. Invertible syntax descriptions og partielle isomorfier (Rendel & Ostermann 2010 + Haskell-økosystemet)

**Mekanisme/API.** Kjernen er `data Iso a b = Iso (a -> Maybe b) (b -> Maybe a)` — et *par* av partielle funksjoner, komponert via tre typeklasser: `IsoFunctor` (`<$> :: Iso a b -> d a -> d b`), `ProductFunctor` (`<*> :: d a -> d b -> d (a,b)` — **tupling, ikke currying**; partielle isoer currier dårlig) og `<|>`, pluss `pure :: Eq a => a -> d a` og `token` ([paper-PDF](https://www.informatik.uni-marburg.de/~rendel/unparse/rendel10invertible.pdf), [Hackage invertible-syntax](https://hackage.haskell.org/package/invertible-syntax)). Iso-algebraen som viste seg *tilstrekkelig* i praksis er liten: `id`/`(.)` (Kleisli-komposisjon over `Maybe`), `×` (produkt-bimap), `associate`/`commute`/`unit` (tuppel-omforming), `element x` (konstant/default; krever `Eq`), `subset p` (identitet innsnevret til et predikat — filtrering), `inverse`, konstruktør-isoer maskingenerert med Template Haskell (`defineIsomorphisms`), og `iterate`/`foldl` (fold omskrevet til småstegs abstrakt maskin som kan kjøres baklengs) — derav `many`, `between`, `chainl1`. To detaljer er spesielt lærerike: (1) `pure`/`element` krever `Eq` fordi *printeren må sjekke at verdien som forkastes faktisk er lik konstanten*; (2) `p <* q`-variantene krever at det ignorerte har type `d ()` — å forkaste ikke-triviell informasjon er forbudt, ellers kan ikke printing rekonstruere den. `subset` på operator-prioriteter gir automatisk korrekt parentes-innsetting ved printing («correct round trip behavior is automatically guaranteed»).

**Adopsjon.** Forskningsgrad: referanseimplementasjonen har eksponentiell backtracking og lekker minne (papirets §4.4 sier det selv); [invertible](https://hackage.haskell.org/package/invertible) (totale bijeksjoner, `<->`/`BiArrow`) har ~5 reverse deps; [codec](https://hackage.haskell.org/package/codec) (applikativ felt-for-felt-kodek `Codec r w a` = deserialiserer + serialiserer) er liten. Idéen *shippet* først via swift-parsing (se §4).

**Veggene.** (a) *Primitive isoer er usjekket*: papiret innrømmer at håndskrevne `Iso`-par «is neither safe nor convenient … it is not checked that the two directions are really inverse» — nøyaktig skillet PySpell-stigen formaliserer. (b) *Kontekst-sensitivitet*: applikativ struktur kan ikke la serialisering avhenge av en parsed verdi (lengdeprefiks!); løsningen er monadiske profunktorer, men da må rundtur-egenskapene sjekkes separat ([Lysxia: Towards monadic bidirectional serialization](https://blog.poisson.chat/posts/2016-10-12-bidirectional-serialization.html)). (c) *Lovene gjelder modulo ekvivalensklasser* (vilkårlig whitespace ↔ ett mellomrom) — strikt `print(parse(x)) == x` holder ikke, bare opp til normalisering; de innførte tre eksplisitte kombinatorer `skipSpace`/`optSpace`/`sepSpace` for å styre dette. (d) *Typeklasse-grensesnittet* ga dårlig ergonomi; lærdommen fra [Lysxia: Better invertible syntax descriptions](https://blog.poisson.chat/posts/2016-10-18-typeclass-interface.html) er å avlede alt fra standard-abstraksjoner og holde den bidireksjonale flaten minimal (kun `token` var genuint retningsspesifikk), og at å kreve `Alternative`/backtracking av alle instanser var en feil.

**Lærdommer for PySpell:** en liten, lukket iso-algebra (komposisjon, produkt, sum-per-konstruktør, `subset`-filtrering, `element`-konstanter, fold/unfold) dekker overraskende mye; konstanter som forkastes må *sammenlignes*, ikke antas, på vei tilbake; informasjon kan aldri droppes uten `unit`-bevis; og «brukerskrevet Iso-par» er den udekkede stigen som må property-testes.

### 2. Binærformat-kombinatorer som faktisk shippet

**binrw (Rust).** `#[binrw]` / `#[derive(BinRead, BinWrite)]` genererer leser *og* skriver fra én spesifikasjon ([binrw.rs](https://binrw.rs/), [docs.rs attributter](https://docs.rs/binrw/latest/binrw/docs/attribute/index.html)). Direktiver deler seg skarpt: *symmetriske fra én annotasjon* — `magic`, endianness, `pad_before/after`, `align`, `assert` (valideres etter les / før skriv), `args`/`import`; *krever begge retninger eksplisitt* — `map`/`try_map` (les-retning og skrive-retning er to lambdas), `parse_with`/`write_with`. Avledede felt løses med `#[bw(calc = expr)]` + `#[br(temp)]`: feltet finnes på wire, **beregnes fra andre felt ved skriving og kastes ved lesing** — binrw sitt svar på residual-felt:

```rust
#[binrw]
struct Msg {
    #[bw(calc = data.len() as u32)] #[br(temp)]
    len: u32,
    #[br(count = len)]
    data: Vec<u8>,
}
```

Bredt brukt i reverse-engineering/modding-miljøer ([GitHub](https://github.com/jam1garner/binrw)). *Lærdom:* skill kombinatorer i «symmetriske av natur» (struktur, konstanter, padding, assertions) og «dual-spec» (vilkårlige mappinger) — og gjør dual-spec-varianten syntaktisk tyngre, så brukeren merker at hen forlater bevist territorium.

**Kaitai Struct — parse-only i ni år.** Serialisering ble bedt om i [issue #27](https://github.com/kaitai-io/kaitai_struct/issues/27) (åpnet 13. sep 2016) og landet først i [v0.11, sep 2025](https://kaitai.io/news/2025/09/07/kaitai-struct-v0.11-released.html), kun for Java/Python, NLnet-finansiert ([prosjekt](https://nlnet.nl/project/Kaitai-Serialization/)). Hvorfor lagget det? Spec-språket tillater *vilkårlige énveis-uttrykk* (`size: len_field * 4 - 2`, `if: not _io.eof`) som kompilatoren ikke kan invertere; designnotatene ([generalmimon-gist](https://gist.github.com/generalmimon/fc22e97faf1fe4b4edc8279b0caa152d), [Serialization Guide](https://doc.kaitai.io/serialization.html)) viser to harde funn: (1) lazy/cachede *value instances* blir stale når en avhengighet settes — de måtte innføre eksplisitt invalidering; (2) størrelser kan ikke utledes automatisk — «`if: not _io.eof` doesn't tell us anything – we can set it both ways and it always satisfies itself». Valgt design: brukeren setter *alle* felt (også avledede) selv, og genererte `_check()`-metoder validerer konsistens før `_write()`, på strømmer med fast størrelse. *Lærdom:* et spec-språk med uinverterbare uttrykk gjør serialisering til et ettermonteringsmareritt; «bruker oppgir, systemet *sjekker*» er retretten når du ikke kan invertere — PySpell bør heller nekte uinverterbare uttrykk i BIDI-modus enn å arve dette.

**construct (Python).** «Declarative and symmetrical parser and builder» ([GitHub](https://github.com/construct/construct)) — trolig den mest brukte bidireksjonale DSL-en i det fri, med operator-overlasting (`"count" / Byte`) og en kontekst-dict som flyter gjennom begge retninger ([context-docs](https://construct.readthedocs.io/en/latest/meta.html)). Det viktige trekantskillet: `Computed` (verdi kun i konteksten, aldri på wire), `Rebuild` (på wire; ved *build* **overskrives** brukerens verdi med en rekomputert fra konteksten; ved *parse* leses den), `Default` (brukerens verdi vinner om oppgitt), `Check` (assertion begge veier):

```python
Struct(
    "count" / Rebuild(Byte, len_(this.items)),
    "items" / Byte[this.count],
)
```

*Lærdommer:* `Rebuild` er residual/ancilla-mønsteret i moden form — avledede felt skal **rekomputeres forward og valideres/leses backward, aldri lagres som sannhet**; en kontekst som er identisk i begge retninger er det som gjør symmetrien mulig; og construct viser at et *dynamisk* språk klarer seg med runtime-feil såfremt feilmeldingen bærer full sti inn i strukturen.

**flat/store vs. codec (Haskell).** [flat](https://hackage.haskell.org/package/flat) og [store](https://hackage.haskell.org/package/store) avleder encoder+decoder *generisk fra typen* — total bijeksjon by construction, men null kontroll over formatet; [codec](https://github.com/chpatrick/codec) gir formatkontroll felt-for-felt, men sjekker ikke at felt-paringen er konsistent. *Lærdom:* «avled fra typen» gir gratis symmetri; kombinator-spesifiserte formater kjøper fleksibilitet mot at konsistens må bevises eller testes — PySpell-stigen bør gjøre begge nivåene eksplisitte.

### 3. Statisk verifisering av inverterbarhet uten avhengige typer

Tre shipped/reelle systemer markerer hva som faktisk er oppnåelig, sortert etter maskineri:

**Regulært domene → avgjørbar typesjekk.** [Boomerang](https://www.seas.upenn.edu/~harmony/) typer streng-lens-kombinatorer (konkatenasjon, union, Kleene-stjerne) med regulære uttrykk og *avgjør* statisk at splitting er entydig (brukt på bl.a. SwissProt; [POPL'08](https://www.cis.upenn.edu/~bcpierce/papers/boomerang.pdf)). [Augeas](https://augeas.net/docs/lenses.html) shippet samme idé industrielt (config-redigering under Puppet m.fl.): hver lens bærer fire regexer (ctype/atype/ktype/vtype), og typesjekkeren bruker automata-biblioteket libfa til å avdekke «vertical/horizontal ambiguities» ([wiki](https://github.com/hercules-team/augeas/wiki/Ambiguities-or-what-do-those-error-messages-from-the-typechecker-mean-%3F)) — men sjekken er så dyr at den er avslått som standard og kjøres som eget test-steg (`augparse`). *Lærdom:* innsnevring av språket (regulært fragment) er det som gjør verifiseringen avgjørbar; veggen er nøyaktig kontekst-sensitivitet (lengdefelt, innrykk). Og: ambiguitets-feilmeldinger fra automata-analyse er notorisk kryptiske — budsjettér for feilmeldings-arbeid.

**SMT på førsteordens kombinator-VCer.** [EverParse/LowParse](https://project-everest.github.io/everparse/) beviser at generert kode er «correct (parsing is the inverse of serialization) and non-malleable (each message has a unique binary representation)» — non-malleability *er* parser-injektivitet — via F* + Z3, og brukes i Microsoft-produkter og for TLS/QUIC ([USENIX'19-paper](https://project-everest.github.io/assets/everparse.pdf)). Arkitektur-poenget for PySpell: hver kombinator bærer sitt eget injektivitets-lemma, og komposisjon *bevarer* egenskapen — spec-forfatteren (3D-frontenden ser ut som C-typedefinisjoner) ser aldri avhengige typer. Injektivitetsbevisene er i praksis førsteordens betingelser (prefiks-frihet, entydige lengder) som SMT knuser automatisk.

**Bevisassistent-enden.** [Narcissus](https://dl.acm.org/doi/10.1145/3341686) (Coq/Fiat, ICFP'19) avleder decoder+encoder fra én formatspec med taktikker «in a way that guarantees that they form inverses of each other», demonstrert ved å bytte ut parsing i mirage-tcpip ([arXiv](https://arxiv.org/abs/1803.04870)). Kraftig, men ekspert-kostnad; ingen praktisk vei for en Python-flate.

**Den praktiske grensen.** By-construction dekker i praksis: komposisjon av primitiver med medbrakte inversbevis, produkt/sum-struktur, domene-innsnevring (`subset`), lineær/relevant variabelbruk (CoreFun-relevans), og lineær aritmetikk (injektivitet av `x*k + c`, `k != 0`, er trivielt SMT-sjekkbar). Ingen av de shippede systemene gjør abstrakt interpretasjon for injektivitet på *generell* kode — alle innsnevrer språket i stedet. Alt utenfor (brukerdeklarerte forward/inverse-par, ikke-lineær aritmetikk, tabelloppslag) håndteres i praksis empirisk: lens-lov-testing med QuickCheck/property-tests er akseptert industripraksis der formelle bevis er for dyre ([Bidirectionalization for the Common People, 2025](https://arxiv.org/pdf/2502.18954), [Olegs lens-lov-notater](http://oleg.fi/gists/posts/2018-12-12-find-correct-laws.html)). Dette *er* PySpell-stigen: strukturbevis der språket tillater det, property-test som eksplisitt merket lavere trinn.

### 4. Kjeding og komposisjons-ergonomi

**Haskell-stilen** (operatorer `<$>`/`<*>`/`<|>`) gir høyreassosierte nestede tupler og typeklasse-feilmeldinger som skremte bort brukere; Lysxias to konkrete grep — minimal retningsspesifikk flate og gjenbruk av standardabstraksjoner — er det som gjorde stilen levelig ([blogg](https://blog.poisson.chat/posts/2016-10-18-typeclass-interface.html)).

**swift-parsing (Point-Free)** er den modne arvtakeren og eksplisitt inspirert av Rendel–Ostermann ([GitHub](https://github.com/pointfreeco/swift-parsing)): parsere komponeres i result-builder-blokker, og *én linjes endring* — `Parse(User.init)` → `ParsePrint(.memberwise(User.init(id:name:isAdmin:)))` — oppgraderer en parser til parser-printer; `Conversion`-protokollen (`apply`/`unapply`, begge kan kaste) er deres partielle isomorfi. Rundtur-kontrakten er dokumentert presist og *toveis*: «For every `input` for which `p.parse(input)` does not throw, `p.print(p.parse(input))` is equal to `input`» og symmetrisk for print-parse ([Roundtripping.md](https://github.com/pointfreeco/swift-parsing/blob/main/Sources/Parsing/Documentation.docc/Articles/Roundtripping.md)). *Lærdommer:* migrerings-ergonomien («samme pipeline, bytt konstruktør») er gull for adopsjon; rundtur-lovene må stå som eksplisitt kontrakt per kombinator; og kompilerings-feil i dypt generiske builder-kjeder er svake punktet — en runtime-verifisert Python-flate kan faktisk gi *bedre* feil ved å peke på nøyaktig hvilket ledd i kjeden som brøt inverterbarheten.

**Python-prior-art for inverterbare pipelines** finnes, men i ML-leiren: [TFP bijectors](https://www.tensorflow.org/probability/api_docs/python/tfp/bijectors/Chain) er en storskala-shippet kombinator-DSL der hver `Bijector` har `forward`/`inverse` (+ Jacobian), `Chain`/`JointMap`/`Invert` komponerer dem, komposisjon er bijektiv by construction — og et input/output-*cache* gjør at `inverse(forward(x))` returnerer `x` eksakt i stedet for å regne flyttalls-invers ([Composition](https://www.tensorflow.org/probability/api_docs/python/tfp/bijectors/Composition)). PyTorchs [`torch.distributions.transforms`](https://github.com/pytorch/pytorch/blob/main/torch/distributions/transforms.py) er samme form: `Transform.inv` er en *førsteklasses verdi*, `ComposeTransform` inverterer ved å reversere og invertere leddene, `bijective`-flagget og `cache_size=1` er del av API-et. [python-lenses](https://github.com/ingolemo/python-lenses) viser optics-komposisjon med `&`-operator og prisms bygget av `wrap`/`unwrap`-par; [bidict](https://bidict.readthedocs.io/) er det trivielle endepunktet (`.inverse`-attributt). *Lærdommer:* i Python vinner objekt-med-`.inverse`-egenskap + eksplisitt `Chain`/`|`-komposisjon over operator-akrobatikk; inversen skal være en verdi man kan sende rundt, ikke en modus; og caching-trikset (husk forward-resultatet, ikke rekompuér inversen) er samme idé som residual-lagring — eksakthet ved rundtur er et *design*-ansvar, ikke en numerisk bønn.

### Lærdommer for PySpell strict-BIDI

- **Iso-algebraen kan holdes liten**: komposisjon, produkt, sum-per-konstruktør, `subset`-filtrering, `element`-konstanter og fold/unfold dekket alt i Rendel–Ostermanns casestudie; ikke design for mer før noen trenger det.
- **Forkastet informasjon må bevises triviell eller sammenlignes**: `pure`/konstanter krever likhetssjekk på vei tilbake (Eq-kravet); dropp av ikke-`unit`-verdier skal være en typefeil i BIDI-modus.
- **Residual/ancilla har et modent forbilde**: construct sitt `Rebuild` og binrw sitt `calc`+`temp` — avledede felt rekomputeres forward og valideres/konsumeres backward; de lagres aldri som autoritativ tilstand. Kaitais `_check()`-design er fallbacken når uttrykk ikke kan inverteres — unngå å havne der ved å nekte uinverterbare uttrykk opp-front.
- **Kaitai-lærdommen i én setning**: et spec-språk som tillater vilkårlige énveis-uttrykk gjør invertering til et niårig ettermonteringsprosjekt; strict-BIDI-dialekten må være restriktiv fra dag én.
- **Verifiseringsstigen matcher shipped praksis**: (1) by-construction for kombinator-komposisjon med per-primitiv inversbevis (EverParse-modellen: hvert ledd bærer sitt injektivitets-lemma, komposisjon bevarer det); (2) avgjørbar sjekk kun i innsnevrede domener (Boomerang/Augeas' regulære fragment; lineær aritmetikk via enkel løser); (3) brukerdeklarerte forward/inverse-par property-testes — akseptert industripraksis, men skal være et *synlig* lavere trinn i API-et.
- **Kontekst-sensitivitet er veggen for applikativ struktur**: lengdeprefiks/avhengige felt krever monadisk (sekvensiell) form, og da må rundtur-egenskapen re-etableres per konstruksjon — planlegg `let`-bundne mellomverdier i IR-en som relevans-sjekkes (CoreFun) i stedet for fri monadisk bind.
- **Ergonomi som beviselig virker i Python**: objekter med `forward()`/`inverse` der inversen er en førsteklasses verdi, pipeline-komposisjon (`Chain`/`|`), én-linjes oppgradering fra énveis til toveis (swift-parsings `Parse` → `ParsePrint`), og runtime-feil som peker på nøyaktig hvilket kjede-ledd som brøt kontrakten (construct-sti-feil > Haskell-typefeil).
- **Dokumentér rundtur-kontrakten toveis og per kombinator** (swift-parsings formulering er malen), og vurder TFP/PyTorch-trikset: eksakthet ved rundtur sikres ved å bære residualet med seg, ikke ved å regne seg tilbake.
## Prior art: inkrementelt view-vedlikehold og stream-prosessering i DB-er

### 1. IVM-motorer: DBSP/Feldera, Materialize, Noria

**Mekanisme.** [DBSP (Budiu et al., VLDB 2023, best paper)](https://www.vldb.org/pvldb/vol16/p1601-budiu.pdf) modellerer databasen som en strøm av snapshots `DB[t]` og endringer som **Z-sets**: funksjoner fra rader til heltallsvekter med endelig support, der negativ vekt = sletting; `ΔDB[t] = DB[t] − DB[t−1]`. Enhver query `Q` får en inkrementell versjon `Q^Δ = D ∘ ↑Q ∘ I` (differensiering ∘ løftet query ∘ integrasjon) med den sentrale garantien at **akkumulert output av den inkrementelle kretsen er identisk med å kjøre `Q` fra scratch på akkumulert input** — og chain rule sier at man inkrementaliserer en sammensatt query ved å inkrementalisere hver delquery uavhengig. Lineære operatorer (filter, map, projeksjon) er sin egen inkrementelle versjon — kostnad ∝ |Δ|; bilineære (join) trenger integrert state: `Δ(a⋈b) = Δa⋈Δb + a⋈Δb + Δa⋈b`. [Materialize](https://materialize.com/blog/strong-consistency-in-materialize/) bygger på differential dataflow med **virtual time**: hver update er en trippel `(data, time, diff)`, alle updates i én kildetransaksjon får samme timestamp, og [alle views er alltid konsistente med input ved ett bestemt virtuelt tidspunkt](https://materialize.com/blog/virtual-time-consistency-scalability/). [Noria (OSDI 2018)](https://pdos.csail.mit.edu/papers/noria:osdi18.pdf) er dataflow-DAG-en med **partial materialization**: operatorer holder bare delvis state, evicted/manglende state rekonstrueres on-demand med **upqueries** tilbake mot base-tabellene — det lazy alternativet i renkultur.

**Smerte/suksess.** DBSP er maskin-sjekket i Lean og krever kun at endringer danner en kommutativ gruppe (updates uttrykkes som retract+insert). [Feldera oppnår exactly-once via periodiske checkpoints + input/output-journaling: recovery laster siste checkpoint, replayer input og forkaster duplisert output](https://docs.feldera.com/pipelines/fault-tolerance/). Materialize argumenterer for at eventual consistency i praksis er «ingen garanti» når endringene aldri stopper — snapshot-konsistens er det essensielle, ferskhet er det forhandlbare. [Noria](https://blog.acolyer.org/2018/10/29/noria-dynamic-partially-stateful-data-flow-for-high-performance-web-applications/) valgte motsatt: kun eventual consistency («hvis skrivingene stilner, konvergerer views»), og betalte i stedet med kompleksitet i upquery-protokollen — upquery-svar kommuterer ikke med in-flight updates, så de måtte garantere at ingen updates var underveis mellom upstream state og join under en upquery, kreve **deterministiske operatorer** og **kommutative merges** i multi-ancestor-operatorer. Resultat: 5× MySQL på Lobsters, ~3× minne-overhead.

**Lærdommer for mpedb.** (1) Delta-modellen bør være retract/insert med vekter — en UPDATE i A er delete+insert i CDC-strømmen; det gjør transformer komposable (B→C) og reversible. (2) DBSP-teoremet er en gratis fuzz-orakel: **resultatet skal være uavhengig av batch-grenser** — fuzz vilkårlig oppdeling av CDC-strømmen i batcher og krev identisk B. (3) Behold Materialize-garantien «B = F(A @ watermark-snapshot)» og ofre ferskhet; ikke Noria-garantien «riktig en gang i fremtiden». (4) Noria viser at lazy flytter all race-kompleksitet inn i upquery-protokollen; deterministiske brukerfunksjoner er en hard forutsetning i alle tre systemene.

### 2. Klassiske in-DB-triggere: PostgreSQL, SQLite sessions, Service Broker

**Mekanisme.** PG-triggere kjører **synkront i skriverens transaksjon**, row-level triggere fyrer per rad (10 000 inserts = 10 000 kall), og [kaskader oppstår når trigger-SQL fyrer nye triggere](https://www.postgresql.org/docs/current/trigger-definition.html). [LISTEN/NOTIFY](https://neon.com/guides/pub-sub-listen-notify) er transient signalering: ingen persistens, tapt hvis ingen lytter (at-most-once), 8000 bytes payload. Kø-i-tabell løses kanonisk med [`FOR UPDATE SKIP LOCKED`](https://www.dbpro.app/blog/postgresql-skip-locked) — atomisk, race-fri jobb-claiming (Que, Oban m.fl. bygger på dette). [SQL Server Service Broker](https://learn.microsoft.com/en-us/sql/database-engine/configure-windows/sql-server-service-broker?view=sql-server-ver17) er transaksjonelle køer i DB-en med «activation» (spawner en stored procedure når meldinger ankommer) — en in-DB-daemon-presedens.

**Smerte/suksess.** [GitGuardians produksjonshistorie](https://blog.gitguardian.com/love-death-triggers/): triggere som prekomputerte aggregater ga usynlige kaskader, nesten ingen spor i logger, og uforutsigbare locking-problemer etter ett år — de migrerte til asynkrone Celery-workers med sekund-nivå lag og konkluderte at triggere må være trivielle og kodegjennomgått som vanlig kode. [Cybertecs benchmarks](https://www.cybertec-postgresql.com/en/are-triggers-really-that-slow-in-postgres/) viser at trigger-*mekanismen* er billig — kostnaden er arbeidet som utføres synkront i skrivebanen. Sterkeste datapunkt mot in-commit-sidearbeid: [recall.ai fant at NOTIFY tar en global eksklusiv lås i commit-fasen som serialiserer ALLE commits](https://www.recall.ai/blog/postgres-listen-notify-does-not-scale) — tre nedetider, CPU/IO stupte mens databasen sto og ventet på én mutex. [SKIP LOCKED-køer i skala](https://richyen.com/postgres/2026/05/04/postgres_job_queue.html) møter MVCC-mismatch: hver claim/complete er en WAL-logget transaksjon, dead tuples, vacuum-press. Service Broker-erfaringen er [gjennomgående «for mange konsepter, umulig å feilsøke»](https://www.sqlservercentral.com/forums/topic/in-praise-of-service-broker).

**Lærdommer for mpedb.** (1) **Evidens mot in-commit-transformer**: i en single-writer-DB forlenger hvert µs transform holdetiden på den globale writer-låsen for *alle* prosesser; PG-verdenen flyktet fra selv billig synkront sidearbeid (recall.ai, GitGuardian). In-commit er kun forsvarlig for validering/CHECK-nivå arbeid og billig CDC-*capture*. (2) Mønsteret alle konvergerer mot er «durabel kø i samme DB + worker claimer atomisk + resultat og cursor committes sammen» — mpedbs CDC-logg + én daemon er degenerert (enklere) SKIP LOCKED: én konsument, null claim-contention. (3) Notify-aktig vekking skal være en ren optimalisering (hint), aldri korrekthetsbærende: daemonen må polle cursoren; at-most-once-signal + durabel logg er tilstrekkelig. (4) Kaskader (A→B→C) må være en eksplisitt, ordnet, dybdebegrenset DAG med kjørbar introspeksjon (cursor, per-funksjon feil som spørrbare tabeller) — ikke implisitte triggere-på-triggere, som er det dokumenterte debugging-marerittet.

### 3. Embedded/serverless-presedenser: sqlite hooks, DuckDB, litestream, cr-sqlite

**Mekanisme.** [`sqlite3_update_hook`](https://www.sqlite.org/c3ref/update_hook.html) fyrer kun **i samme prosess, på samme connection** som gjør endringen, misser WITHOUT ROWID/truncate-optimalisering/REPLACE-konflikt-sletting, og callbacken får ikke modifisere databasen — dvs. applikasjonen må selv drive prosesseringen etter commit; kryssprosess-notifikasjon finnes ikke. [Sessions-extension](https://sqlite.org/sessionintro.html) fanger changesets på skriver-connection og applyer dem med konflikthåndterer på mottakersiden. [DuckDB har ingen triggere](https://github.com/duckdb/duckdb/discussions/12562) — ingen offisiell begrunnelse er publisert, men community-konsensus er at det ikke hører hjemme i OLAP-designet. SQLites eneste «bakgrunnsarbeid», [WAL auto-checkpoint, er kooperativt: det utføres av den committende connection som krysser terskelen](https://sqlite.org/wal.html) — med dokumentert **checkpoint starvation** når det alltid finnes aktive lesere (WAL-en vokser ubegrenset). [Litestream](https://litestream.io/how-it-works/) er en eksplisitt sidecar-daemon som overtar checkpointing via en langlevd read-txn + shadow-WAL; asynkron replikering med [~1 s tapsvindu, og et hardt krav om nøyaktig én replikerende prosess per DB — to instanser gir korrupsjon](https://litestream.io/tips/). [cr-sqlite](https://github.com/vlcn-io/cr-sqlite) splitter: *capture* skjer in-commit (triggere + clock-metadatatabeller skrives inne i skriverens txn), mens *merge/sync* er applikasjonsdrevet pull med lamport-klokker og CRDT-konvergens.

**Smerte/suksess.** Embedded-økosystemets svar på «hvem eier bakgrunnsarbeid» er altså tredelt og mapper nøyaktig på våre opsjoner: **in-commit** (sqlite-triggere, cr-sqlite-capture — funker fordi capture er billig og deterministisk), **eksplisitt daemon** (litestream, mpedbs `mirror pull` — operasjonelt bevist, men fremdrift krever at daemonen kjører), og **kooperativt-på-commit** (WAL-checkpoint — krever ingen ekstra prosess, men stjeler latens fra en tilfeldig writer og sulter under kontinuerlig last).

**Lærdommer for mpedb.** (1) I en no-server-modell kan ikke hooks bære kryssprosess-ETL — **den durable CDC-loggen i delt lager er den eneste pålitelige kanalen**; mpedb har den allerede. (2) cr-sqlite-splitten er riktig: in-commit gjør kun billig, deterministisk capture (CDC-entry); transformen deferres. (3) Velges daemon: håndhev single-instance med lease/lock i shm (litestreams korrupsjons-advarsel; mpedb har robust-mutex + pid-identitet fra før) og **fuzz to daemons som racer**. (4) Spawn-on-attach er skjørt (uklart eierskap, dør med prosessen); en eksplisitt kommando à la `mpedb etl run` — samme modell som `mirror pull` og litestream — er den normaliserte operasjonsmodellen. Kooperativt-på-commit kan i høyden være opportunistisk «top-up», aldri primær fremdriftsmekanisme (starvation-presedensen).

### 4. Cursor/watermark + idempotens: Kafka, Debezium, Flink

**Mekanisme.** [Kafkas exactly-once](https://www.confluent.io/blog/exactly-once-semantics-are-possible-heres-how-apache-kafka-does-it/) hviler på idempotent producer + transaksjoner, der nøkkeltrikset er å **committe consumer-offsets i samme transaksjon som outputen**; `read_committed` skjuler åpne transaksjoner. [Debezium](https://debezium.io/documentation/faq/) persisterer en egen offset-bokmerke inn i kildens change-log; crash ⇒ resume fra sist lagrede offset ⇒ **at-least-once, duplikater er dokumentert normalt og synlige nedstrøms**; [incremental snapshots (DBLog-mønsteret) skriver low/high watermarks inn i loggen rundt chunk-lesing for å avstemme snapshot mot strøm, resumerbart per chunk](https://debezium.io/blog/2021/10/07/incremental-snapshots/). [Flink](https://nightlies.apache.org/flink/flink-docs-stable/docs/concepts/stateful-stream-processing/) checkpointer operator-state **og source-offsets atomisk** via Chandy-Lamport-barrierer; recovery = restore state + replay fra offsets; watermarks er et ortogonalt event-time-kompletthetssignal. Fellesloven: **exactly-once-effekt = at-least-once-levering + idempotent/transaksjonell apply** — ingen leverer «exactly-once delivery».

**Smerte/suksess.** [Offset-tap eller -korrupsjon gir enten replay (duplikater) eller hull](https://risingwave.com/blog/debezium-offset-management-guide/); ekte 2PC-sinks er så dyre at alle i praksis foretrekker idempotente sinks. [Felderas exactly-once](https://docs.feldera.com/pipelines/fault-tolerance/) er samme oppskrift: checkpoint + input-replay + dedup av allerede-produsert output.

**Minimal robust ordning for mpedb**: (1) Daemonen gjør alt i **én destinasjons-write-txn**: apply transform-output for CDC-intervallet `(T_from, T_to]` + sett `cursor = T_to`. SIGKILL hvor som helst ⇒ enten committet (cursor fremme) eller ikke (redo fra samme cursor); redo er trygt fordi apply er nøkkel-idempotent og run-cachen `(input-hash, fn-hash) → output-hash` hopper over rekomputering. Dette er Kafkas offsets-i-output-transaksjonen — og mpedb får det gratis fordi B og cursoren bor i samme DB. (2) Eksponer watermarken som metadata: «B er konsistent per txn T» (Flink-watermark/Materialize-frontier-analogen) — det gjør lazy-aktig lesing mulig som ren policy («vent til watermark ≥ min snapshot» eller «les det som finnes»). (3) Determinisme må håndheves, ikke antas: re-kjør funksjonen på samme input-hash og krev samme output-hash — run-cachen er verdiløs (og konvergens umulig) med ikke-deterministiske funksjoner. (4) Fuzz-listen utover mirror-collide-SIGKILL: **batch-grense-randomisering** (DBSP: resultat må være batch-uavhengig), **to konkurrerende daemons** (lease-brudd), **delete/update-tunge workloads** (retraksjonsbanen, ikke bare inserts), **kill mellom kaskadetrinn B→C**, og som sluttorakel: drain og sammenlign B mot from-scratch `F(A)` — identisk med Felderas garanti og mirror-collide-oraklet.

### Lærdommer for mpedb-triggere

- **Velg daemon som primærmodell**: alle produksjonssystemer (Kafka, Debezium, Flink, Feldera, litestream, GitGuardians retrett fra triggere) konvergerer mot «durabel logg + cursor + idempotent replay». Mirror-daemonen er presedensen; ETL-daemonen er samme maskin med en transform i midten.
- **In-commit kun for capture, aldri transform**: i single-writer-modellen forlenger transform-arbeid writer-lock-holdetiden for alle prosesser; PG-evidensen (recall.ai sin NOTIFY-globallås, GitGuardian-kaskadene) viser at selv «billig» synkront sidearbeid i commit-banen blir serialiseringspunktet.
- **Lazy som komplement, ikke fundament**: Noria beviser at det virker, men all race-kompleksitet flytter inn i upquery-protokollen — og i mpedb kan ikke en lock-fri leser persistere resultater uten å ta writer-låsen. Lazy = policy over watermarken («compute-on-read uten persist», eller les-utløst prioritering av daemonen), ikke en egen skrivebane.
- **Atomisk cursor+output i én txn er hele crash-safety-historien** — gratis når B og cursor deler DB. Ikke bygg noe mer komplisert.
- **Garantien som holdes: B = F(A @ watermark)** (snapshot-konsistens, Materialize); det som ofres: ferskhet. Aldri Noria-style «eventuelt riktig» uten definert snapshot.
- **Delta = retract+insert med fortegn** (Z-set-modellen): UPDATE i A er delete+insert i strømmen; det gjør B→C-kjeder komposable og reversering veldefinert.
- **Determinisme er kontraktsfestet**: run-cache-hashene gir både håndhevelse (re-run, sammenlign output-hash) og gratis dedup ved replay.
- **Wakeup er hint, cursor er sannhet**: polling mot cursoren bærer korrektheten; enhver signalering er kun latensoptimalisering.
- **Ny fuzz-flate utover SIGKILL-collide**: batch-grense-randomisering, dobbel daemon (lease), retraksjonstunge workloads, kill mellom kaskadetrinn, determinisme-recheck — sluttorakelet er alltid drain + from-scratch `F(A)`-sammenligning.
## Prior art: kode lagret i databasen

### 1. Stored procedures (PL/pgSQL, T-SQL, PL/SQL)

**Mekanisme.** Prosedyrer lagres som skjemaobjekter i databasekatalogen og deployes med `CREATE OR REPLACE` — databasen holder bare *siste* versjon, uten historikk. Kall skjer ved navn; logikken kjører i serverprosessen med transaksjonell tilgang til data. Én navngitt prosedyre = én muterbar slot; å endre den endrer oppførselen for alle kallere umiddelbart.

**Adopsjonsrealitet og dokumentert smerte.** Kjernekritikken etter 40 år er påfallende konsistent: (1) *ingen versjonshistorikk* — "if business logic spans across multiple stored procedures then it can be very difficult to establish the exact combination of different versions of different stored procedures at a given point in time" ([dusted.codes](https://dusted.codes/drawbacks-of-stored-procedures)); (2) *utestbarhet* — logikk kan ikke skilles fra data, mocking er umulig, unit-test-forsøk ble stort sett forlatt ([nkdagility](https://nkdagility.com/resources/blog/stop-writing-business-logic-in-stored-procedures/)); (3) *skjult forretningslogikk* utenfor code review/CI, og (4) *vendor lock-in* ([Medium/Binary Notes](https://medium.com/binary-notes/why-using-stored-procedures-is-not-recommended-for-modern-applications-41a8b9c17ba4)). Klassisk drift-feilmodus: en hotfix `CREATE OR REPLACE` rett i prod gjør at miljøene divergerer fra migrasjonshistorikken, med deploy-feil og bugs som bare reproduserer i ett miljø ([Liquibase om drift](https://www.liquibase.com/blog/database-drift), [danielnolan.io](https://danielnolan.io/database-drift-and-migrations/), [odetocode 2008](https://odetocode.com/blogs/scott/archive/2008/02/02/versioning-databases-views-stored-procedures-and-the-like.aspx)). Men de *forble* riktig verktøy for data-nære transformasjoner: set-basert prosessering av millioner rader der data ikke trenger å forlate databasen, integritetsregler som må gjelde uansett hvilken applikasjon som skriver, og korte transaksjoner under motorens kontroll ([yugabyte/dev.to](https://dev.to/yugabyte/triggers-stored-procedures-for-pure-data-integrity-logic-and-performance-1eh8), [codidact](https://software.codidact.com/posts/285745/285752)). Det er nøyaktig PySpells nisje.

**Hvordan moderne team faktisk håndterer versjonering — og konvergensen med content-hash.** Flyway løser sproc-problemet med *repeatable migrations*: prosedyrens tekst ligger i én fil i VCS, og re-appliseres hver gang **checksummen** endres; filnavn + checksum lagres i `flyway_schema_history` ([Redgate-dok](https://documentation.red-gate.com/fd/repeatable-migrations-273973335.html)). Liquibase gjør det samme med `runOnChange="true"` og MD5-checksum i `DATABASECHANGELOG` ([Liquibase-dok](https://docs.liquibase.com/concepts/changelogs/attributes/runonchange.html)). Bransjen har altså allerede *konvergert på checksum-identifisert prosedyrekode* — men som et eksternt lag limt på en muterbar navnslot. mpedbs `etl/<hash>`-register bygger det samme prinsippet inn i selve lagringsmodellen, som fjerner hele drift-klassen: det finnes ingen muterbar slot å drifte.

**Lærdommer for mpedb.**
- Aldri `CREATE OR REPLACE`-semantikk: redigering = ny hash, gamle versjoner beholdes. Drift-feilklassen (hotfix i prod ≠ VCS) kan ikke oppstå når koden er innholdsadressert.
- Tilby `mpedb`-CLI-eksport/-import av PySpell-kilde som *tekstfiler* slik at koden kan ligge i git og reviewes — Flyway/Liquibase-lærdommen er at DB-lagret kode dør uten en friksjonsfri bro til VCS og code review.
- Gjør kall-stedet eksplisitt versjonert: pipelines pinner hash (som planlagt), interaktive kallere resolver navn→hash ved kall. Det gjenskaper "hvilken kombinasjon av versjoner kjørte da?" som sprocs aldri kunne svare på.
- Testbarhet må designes inn: PySpell-funksjoner bør kunne kjøres mot syntetiske rader uten en full database rundt seg (ren IR-evaluator), ellers arver vi sproc-utestbarheten.

### 2. Immutable/innholdsadresserte koderegistre (Unison, Nix, Docker digests)

**Mekanisme.** Unison identifiserer hver definisjon ved en 512-bit SHA3-hash av syntakstreet, etter at navngitte argumenter er erstattet med posisjonelle referanser og alle avhengigheter med *sine* hasher; "names are just separately stored metadata that don't affect the function's hash" ([unison-lang.org/docs/the-big-idea](https://www.unison-lang.org/docs/the-big-idea/)). Navn er pekere inn i et immutabelt adresserom; redigering gir ny hash, og den gamle definisjonen består (bare labelen flyttes) ([SoftwareMill-erfaringsrapport](https://softwaremill.com/trying-out-unison-part-1-code-as-hashes/)). Nix bygger store-paths som hash-av-alle-inputs **+ et menneskelig navn** (`/nix/store/lz9g…-hello-2.12.1`); Docker skiller muterbare *tags* fra immutable *digests*.

**Adopsjonsrealitet og dokumentert smerte.** Unisons gevinster er dokumentert reelle: rename er en ren metadata-operasjon, diamond-dependency-problemet forsvinner (to versjoner sameksisterer som to hasher), typecheck-resultater og testresultater på rene funksjoner caches permanent per hash ([the-big-idea](https://www.unison-lang.org/docs/the-big-idea/)). Smerten er også dokumentert: (1) når en definisjon endres må *dependents* oppdateres — UCM genererer en "todo"-liste og en dedikert update-branch, og dette `update`-flyten er det de har brukt flest år på å polere ([update-dok](https://www.unison-lang.org/docs/ucm-commands/update/), [workflow-dok](https://www.unison-lang.org/docs/usage-topics/workflow-how-tos/update-code/)); (2) rå hasher lekker inn i brukerflaten ("#hmt4gnn927" i scratch-filer) og oppleves kryptisk — å matche gamle hash-referanser mot nye navn er et innrømmet pain point ([SoftwareMill](https://softwaremill.com/trying-out-unison-part-1-code-as-hashes/), [Unison-releases](https://github.com/unisonweb/unison/releases)). Nix-lærdommen: hashen alene er meningsløs for mennesker, derfor bærer hver store-path et navnesuffiks som disambiguator ([Tweag om derivation outputs](https://www.tweag.io/blog/2021-02-17-derivation-outputs-and-output-paths/)). Docker-lærdommen: muterbare tags ga ikke-deterministiske deploys og supply-chain-hull (uendret `node:9`-tag, plutselig ødelagt yarn); digest-pinning er nå anbefalt praksis overalt ([candrews](https://candrews.integralblue.com/2023/09/always-use-docker-image-digests/), [Mend](https://www.mend.io/blog/overcoming-dockers-mutable-image-tags/)) — men *ingen* refererer images ved bare digest; alle beholder tag+digest sammen.

**Lærdommer for mpedb.**
- Lagre navn→hash-bindinger som muterbar metadata over immutabel hashet kode, Unison-style; muter aldri kode in place. Dette er allerede designet — prior art bekrefter at det er riktig kjerne.
- Vis aldri rå hasher som primær identitet: alle CLI/SDK-flater viser `navn@kortversjon` (eller `navn#kort-hash` à la Nix' `hash-navn`), med full hash tilgjengelig for pinning. Unisons største ergonomiproblem var hasher i ansiktet på brukeren.
- Innholdsadressering gir gratis caching: valideringsresultat, kompilert IR og (for rene funksjoner) testresultater kan caches per hash og aldri invalideres — samme egenskap mpedb allerede utnytter for query-planer.
- Ha en `todo`-ekvivalent: når noen publiserer ny versjon av en funksjon andre pipelines avhenger av, må systemet kunne *liste* dependents som fortsatt pinner gammel hash — det er en query, ikke en krise, nettopp fordi gamle hasher aldri slutter å virke.

### 3. Sandboxed brukerkode i moderne databaser

**Mekanisme.** Postgres deler prosedyrespråk i *trusted* (kan gis til vanlige brukere; ingen filsystem/nettverkstilgang, f.eks. plv8) og *untrusted* (kun superuser; PL/Python finnes *bare* som `plpython3u` fordi Python ikke kan sandboxes forsvarlig i prosess) ([postgresql.org/docs/current/plpython](https://www.postgresql.org/docs/current/plpython.html)). SQLite kjører application-defined functions in-process uten sandbox, men krever et eksplisitt `SQLITE_DETERMINISTIC`-flagg for at en funksjon skal få brukes i indekser, generated columns og partial-index-WHERE ([sqlite.org/deterministic](https://sqlite.org/deterministic.html)). WASM-i-DB (libSQL, SingleStore, ScyllaDB) kompilerer UDF-er til WASM og kjører dem i en runtime (typisk Wasmtime): lineær minnemodell med hard cap, ingen ambient I/O (ingen syscalls, filer, nettverk, tråder), og CPU-begrensning via *fuel* (deterministisk instruksjonsbudsjett, ~2x overhead) eller *epochs* (billig timer-basert avbrudd, ikke-deterministisk) ([Wasmtime Config](https://docs.wasmtime.dev/api/wasmtime/struct.Config.html), [Bytecode Alliance om ytelse](https://bytecodealliance.org/articles/wasmtime-10-performance), [ScyllaDB om Wasmtime-valget](https://www.scylladb.com/2022/04/14/wasmtime/)).

**Adopsjonsrealitet og dokumentert smerte.** Trusted/untrusted-skillet er den viktigste organisatoriske lærdommen: uten et trusted-språk blir brukerkode et superuser-privilegium og dør i praksis. SingleStore setter 16 MB minne per Wasm-sandbox som default, utvidbart per funksjon med `GROW TO` ([SingleStore-dok](https://docs.singlestore.com/cloud/reference/code-engine-powered-by-wasm/)). Wasmtime-dokumentasjonen er eksplisitt på at sandboxen hindrer *escape* men ikke *ressurskonsum* — CPU- og minnetak må legges på separat ([systemshardening om epoch-interruption](https://www.systemshardening.com/articles/wasm/wasmtime-epoch-interruption-security/)). Determinisme-siden har to dokumenterte feller: Postgres-funksjoner feilmerket `IMMUTABLE` gir indekskorrupsjon og stale constant-folding i cachede planer ("if you lie to the planner, it will get its revenge") ([xfunc-volatility](https://www.postgresql.org/docs/current/xfunc-volatility.html), [pgsql-hackers](https://www.postgresql.org/message-id/469536.1633969664@sss.pgh.pa.us)); og SQLite-planneren tør bare optimalisere funksjoner som selv deklarerer determinisme ([create_function](https://www.sqlite.org/c3ref/create_function.html)).

**Lærdommer for mpedb.**
- PySpell må være mpedbs *trusted language*: bounded IR uten ambient I/O er nøyaktig egenskapen som lar hvem som helst — ikke bare en admin — lagre og kjøre funksjoner. Ikke lag noen "untrusted escape hatch"; det er dét som tvang PL/Python inn i superuser-gettoen.
- Speil WASM-UDF-grensene eksplisitt i IR-evaluatoren: instruksjonsbudsjett (fuel-modellen — deterministisk, og overheaden er akseptabel for ETL-korn), hard minne-cap per invokasjon med default + per-funksjon-override (à la SingleStores 16 MB/`GROW TO`), og ingen implisitt tilgang til klokke/tilfeldighet/filsystem. Sandbox ≠ ressurskontroll; begge trengs.
- Determinisme må være en *deklarert og håndhevet* egenskap per funksjon, ikke en antakelse. Alt som skal inn i indekser, CHECK-uttrykk eller cachebare resultater krever determinisme-flagget — og siden IR-en er bounded kan mpedb *verifisere* flagget statisk i stedet for å stole på forfatteren, som er hakket bedre enn både SQLite og Postgres.
- Fordi funksjonen er identifisert ved hash, er (hash, params) → resultat en gyldig cache-nøkkel *kun* for deterministiske funksjoner — knytt cachebarhet til det verifiserte flagget.

### 4. Fleretbruker-redigering av DB-lagret kode

**Mekanisme.** Systemer som fungerer eksponerer tre ting: *hvem/når/hvorfor* per endring, en *diff-flate* mellom versjoner, og en *forslags-/merge-mekanisme*. Postgres selv har ingenting innebygd (en `CREATE OR REPLACE` sletter historien); pgAudit bolter på logging av DDL/FUNCTION-klasser med bruker, timestamp og statement ([pgAudit hos Supabase](https://supabase.com/docs/guides/database/extensions/pgaudit)). Migrasjonverktøyene lagrer et minimalt men tilstrekkelig metadatasett per endring: id/beskrivelse, checksum, author, installed_by, installed_on ([Flyway](https://documentation.red-gate.com/fd/repeatable-migrations-273973335.html), [Liquibase](https://docs.liquibase.com/concepts/changelogs/attributes/runonchange.html)). Observable gir hver notebook en historikk-pane med tilbakerulling, og kollaborasjon via "Fork, Suggest, Merge": forslag sendes med en kort *note*, og eieren kan akseptere per celle ved å toggle mellom Parent- og Fork-versjonen ([Observable collaboration](https://observablehq.com/documentation/collaboration/), [history](https://observablehq.com/documentation/notebooks/history)).

**Adopsjonsrealitet og dokumentert smerte.** Smalltalk er 40-års-varselet: kode levde i et image med en changeset-logg som var "barely structured text", et feilutsatt delingsmedium; økosystemet endte med å bygge ekte VCS (ENVY, Monticello), og moderne praksis er "commit source, rebuild images from source" ([HN/Lobsters-tråder om On Learning Smalltalk](https://news.ycombinator.com/item?id=29890205), [Bracha: An Image Problem](https://gbracha.blogspot.com/2009/10/image-problem.html)). Databricks gikk samme vei: innebygd automatisk revision history finnes fortsatt, men legacy notebook-Git-integrasjonen ble fjernet i januar 2024 til fordel for Git folders — den interne historikken ble en bekvemmelighet, git ble sannheten ([Databricks-dok](https://docs.databricks.com/aws/en/notebooks/notebook-version-history), [Microsoft Learn](https://learn.microsoft.com/en-us/azure/databricks/archive/repos/git-version-control-legacy)). Observables lærdom er at brukerne klarte seg med overraskende lite: full versjonshistorikk + fork + et fritekst-notat per forslag + per-celle-diff — ikke branching-galakser.

**Lærdommer for mpedb.**
- Minimalt metadatasett per versjon, immutabelt og lagret ved siden av hashen: `author, timestamp, comment, prev_hash`. `prev_hash`-lenken gjør historikken til en kjede som kan diffes og auditeres — det er nøyaktig feltsettet Flyway/Liquibase konvergerte på, pluss lenken som content-addressering gjør gratis.
- Diff-flaten er kilde-mot-kilde, ikke hash-mot-hash: `mpedb etl diff <navn>@v1 <navn>@v2` må vise PySpell-tekstdiff. Smalltalk-changesets feilet fordi endringsloggen ikke var lesbar som diff.
- DB-historikk er bekvemmelighet, git er sannhet: både Smalltalk og Databricks endte der. Behold in-DB-versjonskjeden for audit og rollback, men gjør tekst-eksport til git førsteklasses (jf. lærdom 1).
- Redigering av andres funksjon bør være forslag, ikke overskriving: siden en edit uansett skaper ny hash, er "suggest" bare *ny hash + notat + prev_hash-peker uten å flytte navnebindingen* — eieren flytter bindingen ved aksept. Observable viser at dette + et kommentarfelt er hele mekanismen brukere faktisk trenger.

### Lærdommer for mpedb (samlet)

- **Immutabilitet er hovedgevinsten**: hele sproc-drift-feilklassen (hotfix ≠ VCS, miljøer divergerer, "hvilken versjon kjørte?") eksisterer bare fordi `CREATE OR REPLACE` muterer en navnslot. `etl/<hash>` + pin-ved-hash eliminerer klassen strukturelt — ikke svekk dette med noen "replace in place"-snarvei.
- **Navn→hash-binding som muterbar metadata over immutabel kode** (Unison-modellen), og vis aldri rå hash som primæridentitet — alltid `navn@versjon/kort-hash` (Nix/Docker: hash *og* menneskelig navn, alltid sammen).
- **Checksum-keyed deploy er allerede bransjestandard** (Flyway repeatable migrations, Liquibase `runOnChange`) — mpedb internaliserer bare mekanismen. Behold broen de bygde den for: friksjonsfri tekst-eksport/-import mot git og code review.
- **Sandbox ≠ ressurskontroll**: bounded IR trenger i tillegg instruksjonsbudsjett (fuel), hard minne-cap med per-funksjon-override, og null ambient I/O (klokke, tilfeldighet, filsystem) — grensene alle WASM-UDF-systemene endte på.
- **Determinisme som verifisert flagg, ikke løfte**: alt som caches, indekseres eller brukes i CHECK krever det, og feilmerking er dokumentert indekskorrupsjon i Postgres. mpedb kan verifisere flagget statisk i IR — gjør det.
- **Per versjon: `author, timestamp, comment, prev_hash`** — mer metadata trengte ingen av systemene; mindre gjorde historikken ubrukelig.
- **Vær trusted language for alle brukere**: ingen untrusted escape hatch. Untrusted-språk blir admin-privilegier og dør (PL/Python-lærdommen).
- **Dependents-spørringen**: ny versjon av en funksjon skal kunne liste hvilke pipelines som fortsatt pinner gammel hash (Unisons `todo`) — trygt nettopp fordi gamle hasher aldri slutter å virke.
