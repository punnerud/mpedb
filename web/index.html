This is not a simulation. The COW B+tree, MVCC snapshots, the freelist commit fixpoint, rigid type and CHECK validation, the MPEE join solver and content-hashed plans are all the same Rust code that runs natively — compiled to WebAssembly. Only the kernel is emulated: mmap is a byte array, msync is a no-op, and the writer lock is a flag, because a browser tab is one process with one thread and no durability.

Everything below runs locally in this tab. Nothing is sent anywhere, and there is no server.

What this page cannot show you

mpedb's headline property is multi-process concurrency: several OS processes writing the same file at once, any of which may be SIGKILLed mid-commit without corrupting it. You are not seeing that here, and it cannot be faked in a browser.

Why not: the :memory: path this page runs is process-private by construction — which is exactly why it ports so cleanly. Two browser tabs are two WebAssembly instances with two separate byte arrays; they share nothing, so there is no second writer to show.

It is not impossible, just not built yet. SharedArrayBuffer plus Web Workers would give several independent contexts writing the same bytes, and Atomics.wait/notify and Atomics.compareExchange are genuine futex and CAS primitives — enough for the reader-pin protocol and the writer wait queue. The missing piece is owner-death detection: the browser has no robust mutex, no EOWNERDEAD, and no flock the kernel releases when a holder dies. That is the mechanism the whole crash-safety argument rests on, and it would need a third lock construction with its own design review before this page could claim consistency under a killed writer. Sketched in DESIGN-WASM-MULTIWRITER.md; deliberately not improvised into a demo.

Also absent here, for the same reason: durability classes (commit/wal/async — the in-memory build refuses anything but none, so no group commit and no intent ring), crash recovery and the power-loss simulation, opening a real .db/.mpedb file (there is no filesystem), and the libsqlite3-compatible C-API shim. Those are measured and tested natively — see the README and the engine comparison, where the multi-writer and durability numbers live.

What is genuine on this page: the SQL semantics, the query results, the plans and their content hashes, the precomputed footprints, the MPEE join reordering, and every refusal.
