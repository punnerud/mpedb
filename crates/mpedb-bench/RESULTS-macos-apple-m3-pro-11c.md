>

> mpedb-bench` and can say what that promises: *every cell measured on one
> machine, in one run, by one loop.* **This one cannot.** It is transcribed from
> the 2026-07-14 M3 session, where the cells were collected as **separate
> `--only mpedb` and `--only sqlite` invocations** — at the time the harness
> wrote to a single fixed `RESULTS.md`, so a second engine's run overwrote the
> first's and no combined file could survive.
>
> So read it as a **record of findings**, not as a generated report:
> - cells come from different invocations, so cross-engine ratios carry
>   run-to-run variance that a real single-run report does not;
> - **PostgreSQL is absent** — it was never installed on that machine;
> - the none-class medium is a **RAM disk (APFS over RAM), not tmpfs** — macOS
>   has none. mpedb is mmap-based and pays a filesystem layer SQLite's
>   read/write path does not, so **none-class ratios here are not comparable to
>   the Linux run.**
>
> Replacing this with a genuine single-run report needs PostgreSQL installed on
> that machine and one full `cargo run --release -p mpedb-bench`. Until then the
> honest thing is to say so at the top rather than let the filename imply

## none-class, ops/s (RAM disk — see the caveat above)

| op | mpedb | SQLite | ratio |
|---|--:|--:|--:|
| point-select (PK) | **1,699,066** | 327,907 | 5.2× |
| point-insert | **196,931** | 117,072 | 1.7× |
| point-update (PK) | **246,983** | 140,530 | 1.8× |
| contended-writes (4 threads) | **132,992** | 101,975 | 1.3× |
| read-while-write (reads) | **3,655,502** | 178 | — |

The write ratios **narrow** against Linux (1.7× here vs 3.97× there on insert).
Some of that is the APFS-over-RAM medium; **we did not isolate how much**, so do
not read it as a code signal either way.

That last row is not a typo. SQLite's none-class journal serializes readers
against the writer, and with 11 cores the writer (86,580 writes/s) starves them:
**178 reads/s, p99 139 seconds.** mpedb's MVCC readers are untouched at 3.7M/s.
A pathological config rather than a fair fight — but it is the exact failure mode
mpedb's design exists to avoid, and more cores make it worse, not better.

## commit-class: the durability trap this machine exposed

This is why the M3 run was worth doing, and it is not about throughput. macOS's
`fsync()` does **not** flush the drive's write cache — only
`fcntl(F_FULLFSYNC)` does. Both engines were getting that wrong, in the same
direction: *reporting a write as durable when it was not.*

Single-client durable INSERT, before and after making both honest:

| | ops/s | p50 | really durable? |
|---|--:|--:|---|
| SQLite `synchronous=FULL` (harness default) | 26,642 | 25 µs | ❌ |
| mpedb `durability=commit` (before fix) | 7,583 | 127 µs | ❌ |
| **SQLite FULL + `PRAGMA fullfsync`** | **286** | 3,815 µs | ✅ |
| **mpedb `durability=wal`** | **293** | 3,091 µs | ✅ |
| **mpedb `durability=commit` (after fix)** | **142** | 6,999 µs | ✅ (two flushes) |

**~290 ops/s is simply what an Apple SSD platter flush costs.** Everything above
it was an engine skipping the flush. The honest verdict: mpedb `wal` and
SQLite-with-fullfsync are **tied** (293 vs 286); the 93× and 26× "wins" either
engine appeared to have were artifacts of the measurement, not properties of the
engines.

Both bugs are fixed (`eng_sqlite.rs::fullfsync_on_apple`, and `F_FULLFSYNC`
after `msync` in `shm::msync_range` on non-Linux). mpedb `commit` still pays
**two** flushes (data range + meta page); `F_FULLFSYNC` is per-fd, not
per-range, so one barrier before the ack would do — but that is reviewed
commit-path code (DESIGN.md §5) and is logged as known-issue #0 in
[`BENCHMARKS.md`](../../BENCHMARKS.md).
